K8s Practice: Deploy Your Project to Minikube
Can you take every service your group designed and run it correctly in Kubernetes — with config separated from code and credentials never in plain text?
Days 12–13 taught K8s objects on toy apps. Today those objects meet the real project your group has been designing since Day 8. The non-negotiable rule for today: every environment variable lives in a ConfigMap or a Secret — nothing hardcoded in the Deployment YAML. In production, hardcoded values mean environment-specific image builds, credentials in version control, and no way to rotate a secret without redeploying. Getting this right now means your agent and your pipeline both work correctly from Day 15 onward.
Group day — work with your project partner
Both partners deploy today using the shared docs repo as the spec. Read your DOP and infrastructure doc before writing any YAML. Both of you should end Day 14 with every project service Running in your own minikube cluster.
Setup — same minikube cluster, ingress addon still enabled
minikube status  ·  kubectl get nodes  ·  minikube addons list | grep ingress
Discovery chain — 8 questions
1
Before writing any YAML: list every service from your DOP and build a deployment checklist — one row per service, classifying every env var as ConfigMap or Secret.
My project has these services: [list from your DOP]. For each service, give me a checklist table with: (1) image format, (2) container port, (3) all env vars the service reads at startup, (4) classify each env var — ConfigMap or Secret using this rule: "Would this value cause a security incident if it appeared in a CI log or a PR diff?" No → ConfigMap. Yes → Secret. Examples: DATABASE_HOST → ConfigMap, LOG_LEVEL → ConfigMap, DATABASE_PASSWORD → Secret, JWT_SECRET → Secret, (5) Service type: ClusterIP or needs Ingress route, (6) needs PVC? After the table: total number of K8s objects I need to create. Give me that count before I write a single line of YAML.
The env var key names matter — your ConfigMap keys must match exactly what the application code reads from the environment. A typo here causes CrashLoopBackOff with a confusing error.
Create a k8s/ directory in your repo now. Naming: api-cm.yaml, api-secret.yaml, api-deployment.yaml, api-svc.yaml — one file per object.
2
Write the ConfigMaps for your project's services. Choose between envFrom (whole ConfigMap as env vars) and env.valueFrom.configMapKeyRef (one specific key). Apply all of them.
Based on the checklist, help me write ConfigMap YAML for my project services. (1) Show both Deployment reference methods side by side: Method A: envFrom.configMapRef — injects every key as an env var. One line in the Deployment. Use when you want all values and naming is consistent. Method B: env.valueFrom.configMapKeyRef — pulls one key, lets you rename it. Use when you need to pull from multiple ConfigMaps selectively. (2) Use stringData (not data) — human-readable, no manual base64 needed. (3) Write the ConfigMap for [my first service]. Apply: kubectl apply -f api-cm.yaml. Verify: kubectl get configmap api-cm -o yaml. (4) Critical: the ConfigMap metadata.name must match exactly what envFrom.configMapRef.name references. Show the error message when the name is wrong — kubectl describe pod [pod] shows "configmap not found" in Events. (5) Write and apply ConfigMaps for ALL remaining services. kubectl get configmaps should list all of them.
Use stringData in your ConfigMap, not data. Both store the same way internally — stringData is just easier to write and review in a PR without base64 decoding everything.
3
Write the Secrets for your services. The rule is strict: real credential values must not appear in any YAML file committed to Git. Use kubectl create secret instead.
Handle the sensitive values from the checklist. (1) Why you cannot commit real Secret values to Git: git history is permanent — even deleting the file doesn't remove the value from history. A private repo today can become a public leak tomorrow. (2) Safe creation — use kubectl create secret, not YAML: kubectl create secret generic api-secret \ --from-literal=DATABASE_PASSWORD=dev-password-123 \ --from-literal=JWT_SECRET=dev-jwt-secret The real value never touches a file on disk. (3) Quick reinforcement: base64 in Secret YAML is encoding, not encryption. kubectl get secret api-secret -o yaml, then base64 -d reveals the value instantly. What actually protects a Secret is RBAC. (4) Two mount methods in a Deployment: env.valueFrom.secretKeyRef — value becomes env var. Simple. Shows up in `kubectl exec [pod] -- env`. volumeMount with secret volume — value mounted as file in tmpfs. Doesn't appear in env output. Preferred for TLS certs and long tokens. For today: use env.valueFrom.secretKeyRef for passwords and API keys. (5) Create Secrets for ALL services via kubectl create secret. Verify: kubectl get secrets.
For this training, use realistic dummy credentials like dev-password-123. The habit that matters is knowing WHERE the value comes from — never a YAML file in Git.
In a real project, credentials come from a secrets manager (AWS Secrets Manager, HashiCorp Vault). The kubectl create secret workflow is the dev approximation — the source of truth is external, not in the repo.
4
Deploy your first service: write the Deployment (referencing ConfigMap + Secret) and Service. Apply both, then prove the wiring works by reading the injected values from inside the running Pod.
I have the ConfigMap and Secret for [first service]. Write and apply the Deployment and Service: (1) Deployment spec — include: envFrom.configMapRef: pointing to the ConfigMap name env.valueFrom.secretKeyRef: for each Secret key resources.requests: cpu 100m, memory 128Mi A livenessProbe if the service has a /health endpoint. Apply order: ConfigMap and Secret must exist before the Deployment. If not, Pod enters CreateContainerConfigError. (2) ClusterIP Service — targetPort must match the container's listen port exactly. (3) Three verification commands in sequence: kubectl get pods -w (watch state transitions — Running 1/1 is success) kubectl logs [pod] --tail=30 (look for startup message, no error stack trace) kubectl exec -it [pod] -- env | grep DATABASE (confirm ConfigMap values injected) (4) CrashLoopBackOff: kubectl logs [pod] --previous shows the crashed container's output. The error is almost always in the last 10 lines. (5) If env vars are present but wrong value: kubectl apply -f api-cm.yaml to update, then kubectl rollout restart deployment [name] to force Pods to pick up new values.
The kubectl exec [pod] -- env | grep DATABASE command is the definitive proof of wiring. Do not skip this — seeing the value injected confirms the entire ConfigMap → Deployment → Pod chain is working.
5
In Docker Compose your services found each other by service name (e.g. http://database:5432). In Kubernetes, what is the correct hostname — and how do you update your ConfigMaps to use it?
Explain Kubernetes in-cluster DNS and help me update all ConfigMaps: (1) Format: [service-name].[namespace].svc.cluster.local. Within the same namespace, just [service-name] works. The service-name is the metadata.name in your Service YAML. (2) For each service that depends on another: map the old Docker Compose hostname to the new K8s Service name. Give me that mapping as a table. (3) Test DNS from inside a running Pod: kubectl exec -it [pod] -- nslookup [service-name]. Successful response shows the ClusterIP. NXDOMAIN means the Service doesn't exist or has a different name. (4) After updating ConfigMaps: kubectl apply -f [svc]-cm.yaml, then kubectl rollout restart deployment [name]. (5) Race condition: Pod A starts before Pod B's Service is ready. Simplest dev workaround: configure retry/reconnect in the app code, or use an init container that loops nslookup until the dependency resolves.
The Service name is what other Pods use as hostname — name your Services clearly: api-svc, postgres-svc. Avoid generic names like service when you have six of them.
6
Deploy all remaining services. For each one: ConfigMap → Secret → Deployment → Service, in that order. Troubleshoot any startup failures. All Pods must reach Running 1/1.
Walk me through a repeatable deploy-and-verify loop for my remaining services: (1) Apply order per service: kubectl apply -f [svc]-cm.yaml → kubectl create secret generic [svc]-secret ... → kubectl apply -f [svc]-deployment.yaml → kubectl apply -f [svc]-svc.yaml. (2) After each Deployment apply, watch for 30 seconds: kubectl get pods -w. The four bad states and causes: Pending: insufficient resources — lower resources.requests ImagePullBackOff: wrong image name/tag — fix image field CrashLoopBackOff: app crashes on start — kubectl logs [pod] --previous CreateContainerConfigError: ConfigMap or Secret name mismatch — kubectl get cm,secret and compare names (3) After each service is Running: kubectl exec -it [pod] -- env | grep [a key from its ConfigMap]. (4) Deploy in dependency order: databases and caches first, then backend services, then gateway/frontend. Your architecture diagram shows this order.
Work one service at a time. Deploying everything at once and debugging multiple CrashLoopBackOffs simultaneously is far harder than fixing them one by one.
7
All services are Running. Write the Ingress to route external traffic to the right backends. Test every path with curl.
Write and apply the Ingress for my project: (1) My routes: [fill from DOP — e.g. /api → api-svc:3000, / → frontend-svc:80]. Write full Ingress YAML with pathType: Prefix. (2) Access from outside minikube: Option A: minikube tunnel (run in separate terminal) → curl -H "Host: myproject.local" http://127.0.0.1/api/health Option B: curl -H "Host: myproject.local" http://$(minikube ip)/api/health (3) Test every path. Diagnose errors: 502 Bad Gateway: backend Service has no healthy Endpoints → kubectl get endpoints [svc-name] shows → Service selector doesn't match Pod labels. 404 from nginx: path rule didn't match → check pathType and exact path string. (4) Verify Ingress routes to Service name not Deployment name. backend.service.name in Ingress must match metadata.name in the Service YAML exactly.
kubectl get endpoints is the fastest debug tool for 502 errors. If a Service shows <none> Endpoints, compare the Service selector with kubectl get pods --show-labels — the labels must match exactly.
8
The full stack is Running. Make one real end-to-end API call. Then write memory/kubernetes-project.md — your agent's reference for every future K8s task on this project.
Two tasks to close Day 14: Task 1 — End-to-end verification: Make an API call that exercises real application logic — create a resource, read it back, confirm the response. This proves the full chain: Ingress → Service → Pod → application code → database. Then: kubectl get all — what does a fully healthy stack look like for each resource type? Task 2 — Write memory/kubernetes-project.md: 1. Table: all services, their K8s Service name, namespace, port 2. ConfigMap vs Secret rule — one sentence, your version from today 3. In-cluster DNS pattern for this project (e.g. api-svc.default.svc.cluster.local) 4. Apply order: which service must be Running before others can start 5. Failures hit today + what fixed them — most valuable for future sessions Format: tables and short bullets. No prose. This file is loaded into agent context, not read by a human.
The apply-order entry is the most likely thing you will re-debug. Write it now: "postgres-svc must be Running before api-svc, api-svc before gateway." Your agent follows this on every future deploy without rediscovering it.

Helm Intro: Install Your Project's Dependencies
Your project needs Redis or RabbitMQ. Writing those manifests by hand is 70+ lines you now own forever — and the operator's knowledge that goes into configuring them correctly is not in any YAML tutorial. What does Helm give you instead?
Day 14 got your own services running. But your project design already calls for Redis, RabbitMQ, or PostgreSQL — infrastructure services with complex production requirements that have nothing to do with your application code. Helm is the package manager for Kubernetes. A chart is a package. A release is an installed instance. One install command gives you a StatefulSet, PVC, Service, and ConfigMap — battle-tested defaults, the operator's production experience baked in. Today you watch the intro video, understand the chart model, and use Helm to install the 3rd-party dependencies your project needs.

What to look for #1
How helm install creates a named release stored in the cluster as a Secret — this is what enables history and rollback. Helm is tracking the deployment as a unit, not just applying YAML.
What to look for #2
How values.yaml and --values override files work together. The precedence: rightmost file wins, then --set wins over all files. This is the pattern you'll use to customize 3rd-party charts.
Discovery chain — 8 questions
1
Your project needs Redis (or RabbitMQ, or PostgreSQL). What would deploying it to Kubernetes by hand actually require — and why does that become unsustainable?
My project needs [Redis / RabbitMQ / PostgreSQL]. What would deploying it manually look like? (1) List every K8s object it needs. For Redis: StatefulSet (not Deployment — why StatefulSet for stateful apps?), Service, PersistentVolumeClaim, ConfigMap for redis.conf, Secret for AUTH token. Estimate total lines of YAML. (2) Redis has a production gotcha: if you don't set maxmemory and maxmemory-policy in redis.conf, Redis consumes all available node memory and gets OOMKilled. How would you know to configure this writing YAML by hand vs using a Helm chart where an operator already built it in? (3) What happens when Redis 7.2 releases a security patch — which files change, and what's the risk of doing it wrong? (4) What Helm gives you instead: one install command, battle-tested defaults, operator knowledge baked in as values. (5) What Helm does NOT solve: your own application code, data migration between major versions, backup strategy. Helm installs the service — you still own the operational decisions.
StatefulSets give each Pod a stable network identity (redis-0, redis-1) and a stable PVC — because Redis replicas have roles (primary vs replica) and data must be tied to a specific Pod, not float between them.
2
Run helm create myapp. Explore the generated structure. How does a value in values.yaml flow through to a {{ .Values.image.tag }} reference in a template?
Walk me through the chart structure helm create generates: (1) List and explain every generated file: Chart.yaml, values.yaml, templates/deployment.yaml, templates/service.yaml, templates/ingress.yaml, templates/_helpers.tpl, templates/NOTES.txt, .helmignore. (2) Open values.yaml. Show image.tag. Then show the {{ .Values.image.tag }} reference in templates/deployment.yaml. Explain the dot-notation path: .Values is root, image is nested key, tag is field. (3) What is _helpers.tpl? Why does the leading underscore matter? What is a named template ({{- define "myapp.fullname" -}})? (4) Run: helm template . to render with defaults. Then: helm template . --set image.tag=v2.0.0 — show the difference. (5) What happens if you reference a key that doesn't exist in values.yaml — error or empty string? Key takeaway: helm template . --debug is your dry-run before every helm install. Render first, apply second.
helm template . --debug renders the full YAML locally with template file names above each block. Run this before every helm install to catch template errors without touching the cluster.
3
Add the Bitnami repository. Find the chart for your project's 3rd-party service. Inspect its values — and identify the three keys you must override before installing.
Walk me through adding a Helm repo and inspecting a chart before install: (1) Add and update: helm repo add bitnami https://charts.bitnami.com/bitnami helm repo update helm search repo [service-name] What does the output show — chart name, chart version, app version, description? (2) Inspect the chart's values: helm show values bitnami/[chart-name] This outputs 300–600 lines. Navigate efficiently: helm show values bitnami/redis | grep -A 10 "auth:" helm show values bitnami/redis | grep -A 5 "persistence:" helm show values bitnami/redis | grep -A 5 "resources:" (3) For my project I need to override: (a) auth credential, (b) persistence storage size, (c) resource limits suitable for dev. Show me those key names in the chart. (4) Write a minimal my-[service]-values.yaml with only those three overrides. Everything else uses the chart default — that is the point. (5) Why NOT to use --set for the password: it appears in shell history and in the Helm release Secret stored in etcd.
Run helm show values bitnami/redis | wc -l to see how many configurable values exist — usually 400+. You override the handful that differ from defaults. The rest is production knowledge you get for free.
4
Install the chart using your overrides file. Read the post-install NOTES carefully — they contain the exact service name your application must use to connect.
Install [Redis/RabbitMQ/other] for my project: (1) Install command: helm install my-redis bitnami/redis -f my-redis-values.yaml --namespace default What does Helm print after a successful install? (The NOTES.txt — read every line. It contains the hostname and the connection test command.) (2) Verify running: kubectl get pods, kubectl get svc. What is the Service name Helm created? Write it down — this is the hostname your app uses. (3) The service name follows the pattern: [release-name]-[chart-component]. For helm install my-redis bitnami/redis, the master service is my-redis-master. The NOTES tells you this. Do not guess. (4) helm list — what columns does it show? What does the STATUS column value "deployed" mean vs "failed"? (5) If the install fails or a Pod is Pending: kubectl describe pod [helm-pod-name] — Events section shows the reason. Most common in minikube: PVC pending because the default StorageClass doesn't exist. Fix: add --set persistence.enabled=false for dev.
The NOTES.txt Helm prints is not decoration. It contains the exact connection string and a test command. Trainees who skip reading NOTES spend 20 minutes guessing the service name that was printed right in front of them.
5
Update your application's ConfigMap with the Helm service name. Restart the affected Deployment. Verify the connection from inside your app Pod.
Connect my application to the installed chart service: (1) Which env var in my application's ConfigMap holds REDIS_HOST (or AMQP_URL, or DATABASE_HOST)? Change its value from the old Docker Compose hostname to the Helm service name from Q4. (2) Apply the updated ConfigMap: kubectl apply -f [app]-cm.yaml. Then force the Deployment to pick it up: kubectl rollout restart deployment [app-name]. Watch: kubectl rollout status deployment [app-name]. (3) Test the connection from inside the app Pod: For Redis: kubectl exec -it [app-pod] -- redis-cli -h my-redis-master ping → should return PONG For RabbitMQ: kubectl exec -it [app-pod] -- [connection test command from your app] For PostgreSQL: kubectl exec -it [app-pod] -- psql -h my-postgresql -U [user] -c "\l" (4) If the connection fails: check kubectl get svc | grep redis — is the service Running? Check kubectl exec [app-pod] -- nslookup my-redis-master — does DNS resolve? (5) Once connected: kubectl get all should show both your project services and the Helm-installed service Running side by side. Take a screenshot for the report.
After kubectl rollout restart, watch the new Pods come up: kubectl get pods -w. You should see one old Pod terminate as one new Pod reaches Running — the env var update happened cleanly.
6
Explore the Helm release model: what does a release track, what is a revision, and what does helm history show?
Explain the Helm release model using my installed release: (1) Run: helm list. What columns appear — name, namespace, revision, status, chart, app version. What does a revision of 1 mean? (2) Run: helm history my-redis. What does each row show — revision number, timestamp, status, chart version, description. How is "install" shown differently from "upgrade"? (3) Where does Helm store this history? (Answer: as Kubernetes Secrets in the same namespace — run kubectl get secrets | grep helm to see them.) Why does this mean Helm releases survive pod restarts? (4) The key insight: helm rollback my-redis 1 reverts ALL objects in the release atomically — StatefulSet, Service, ConfigMap, PVC references — as one unit. Compare to kubectl rollout undo which only reverts a Deployment. (5) Demonstration: make a small change to my-redis-values.yaml (e.g. change a label), run helm upgrade my-redis bitnami/redis -f my-redis-values.yaml, then helm history my-redis — confirm revision 2 appears.
helm upgrade --install my-redis bitnami/redis -f values.yaml is the idempotent form — installs if release doesn't exist, upgrades if it does. This is the form used in CI/CD pipelines.
7
Run a real helm upgrade — change a value, verify the revision increments. Then run helm rollback and confirm the cluster reverts.
Practice the helm upgrade → rollback workflow: (1) Make a meaningful change to my-redis-values.yaml — e.g. change resources.requests.memory from 128Mi to 256Mi. Run: helm upgrade my-redis bitnami/redis -f my-redis-values.yaml. Verify: helm history my-redis shows revision 2 or higher. (2) Check the change applied: kubectl describe statefulset my-redis-master | grep -A 5 "Limits" — confirm the new resource value is present. (3) Simulate a bad upgrade: change a value to something invalid (e.g. a non-existent image tag). Run helm upgrade. What happens — does Helm roll back automatically, or does the old release stay running? (4) Run: helm rollback my-redis [previous-revision]. Then helm history — rollback creates a NEW revision, it does not rewrite history. Confirm the cluster is back to the previous state. (5) When to rollback vs fix-forward: rollback immediately if a production health check fails post-upgrade. Fix-forward (new upgrade with corrected values) when you have time to investigate properly. Never leave a release in FAILED status in production.
Helm rollback creates a new revision rather than rewinding history — so helm history always shows the full audit trail: install → upgrade → rollback. This is intentional — you should always be able to see what happened.
