Linux Basics
When a production server goes down at 2am and you SSH in to a blank terminal — what do you actually do?
Linux is the operating system running almost every server, container, and cloud VM. When something breaks in production, you are staring at a bash prompt with no GUI and no installer wizard. Everything a DevOps engineer does — deploying containers, managing services, reading logs, fixing permissions — happens at the Linux command line. A DevOps engineer who is not fluent in the Linux CLI is blocked at every step of every real incident.

Before you start — set up your Linux terminal
⚠ This is a hands-on day. You need a real Linux terminal — not a browser simulator.
Commands like systemctl, chown, and find must actually run and return real output. Reading Claude's answers without running the commands does not count as completing this day.
Option 1 — WSL2 (recommended for Windows)
WSL2 gives you a real Ubuntu terminal inside Windows. Run once in PowerShell (as Administrator):
wsl --install Restart your computer. Open "Ubuntu" from the Start menu. Done — you have a real Linux shell.
If WSL is already installed: just open the Ubuntu app or type wsl in any terminal.
Option 2 — Killercoda (zero setup, browser-based)
Go to killercoda.com → search "Ubuntu Playground" → launch. You get a full Ubuntu VM in your browser — systemctl works, apt works, everything works.
Use this if you can't install WSL2 or are on a managed laptop.
Already on Mac or Linux? You're ready — just open Terminal. On Mac, note that brew replaces apt and systemd is not available — use the Killercoda playground for the systemd questions.
Once your Linux terminal is open: launch Claude Code in your proops2026/ folder as usual. CLAUDE.md + Notion MCP load automatically. Keep both windows side by side — Claude on one side, your Linux terminal on the other. Copy each prompt into Claude, run the commands it gives you, then move to the next question.
Discovery Chain — 8 questions, one incident scenario
Today's 8 questions follow a single production incident from start to finish. Q1 is you SSHing in. Q8 is you writing the memory file after everything is resolved. Copy each purple prompt, paste into Claude, run the commands it gives you, understand the output — then move to the next question.
1
You SSH into a production server that's behaving strangely. What commands tell you where you are — hostname, OS version, current user, and how long the system has been running?
I just SSH'd into a production server that's behaving strangely. Before I do anything else, I need to understand where I am. Give me the exact commands to check: (1) the hostname, (2) the OS version and kernel, (3) which user I'm running as and whether I have sudo, (4) how long the system has been up and the current load. Show me the commands and explain what each line of output means.
Run these commands for real in your terminal — see your own output
2
You need to find the application's log file — you know it's somewhere under /var but not the exact path. How do you find it, and how do you search it for a specific error string?
I need to find an application log file somewhere under /var but I don't know the exact path. Show me: (1) the command to find files by name pattern across a directory tree (e.g., find any file named "app.log" or "*.log" under /var), (2) how to search inside a log file for lines containing a specific error string without opening the whole file, (3) how to show the last 50 lines of a log file and watch it in real time as new lines appear. Give me the exact commands with real examples.
Key commands: find, grep, tail -f — you will use these every on-call shift
3
The log shows a permission error — the app can't write to /var/app/data. You run ls -la. How do you read the permission string, and what is the minimum fix without making the directory world-writable?
The application log shows: "permission denied: cannot write to /var/app/data". I run ls -la on that directory and see: drwxr-xr-x 2 root root 4096 Apr 5 09:00 data. Explain to me: (1) how to read that permission string character by character — what each position means, (2) who can currently read, write, and execute on this directory, (3) the exact chmod command to give the app write access with the minimum permission change — and why making it 777 is wrong. Show me the numeric and symbolic chmod syntax.
Understand this: 755, 644, 700 — what each means and when to use it
4
The app runs as user appuser but the directory is owned by root. How do you change ownership — and why is running applications as root a hard no?
The application runs as "appuser" but /var/app/data is owned by root, which is why it can't write there. Show me: (1) the exact chown command to change the owner and group of that directory to appuser, (2) how to verify the change worked with ls -la, (3) a clear explanation of why running an application process as root is a security problem I should refuse — what specifically can go wrong if the app is compromised while running as root versus running as a limited user.
Least privilege: the single most important Linux security concept for DevOps
5
The application is a systemd service and it crashed. What commands show you the status, restart it, check its logs, and make sure it auto-restarts if it crashes again?
The application is managed by systemd as a service called "myapp". It crashed and is not running. Show me the exact commands to: (1) check the current service status and see the last error, (2) restart the service, (3) view the last 100 lines of the service's logs using journalctl, (4) configure the service to automatically restart if it crashes — show me where in the systemd unit file this is set and what the Restart= options mean. Also explain the systemd service lifecycle: what happens from "systemctl start" to the process actually running.
systemd is how almost every production process runs — learn this cold
6
The server is slow. You suspect high CPU or memory. What three commands give you a live view of what's happening — and how do you kill the process consuming 100% CPU?
The server feels slow after I restarted the service. I need to diagnose it. Show me: (1) the three commands to get a live view of running processes, CPU usage, and memory usage — explain the key columns in each output and what numbers should concern me, (2) how to identify the process consuming the most CPU from that output, (3) the exact command to kill that process — explain the difference between kill (SIGTERM) and kill -9 (SIGKILL) and when to use each. Also show me how to find which process is listening on a specific port (e.g., port 8080).
ps, top, ss — diagnostic before action, always
7
You need to install a monitoring tool the server doesn't have. What is the two-command sequence — and why must the update command run first?
I need to install "htop" on this server for better process monitoring. Show me: (1) the two-command sequence to install a package on a Debian/Ubuntu server — and explain exactly why running the update command first is required, what breaks if you skip it, (2) the equivalent commands on a RHEL/CentOS/Amazon Linux server, (3) how to verify the package was installed successfully, (4) how to list all installed packages and search for a specific one. Keep it practical — what I'd actually run during an incident.
apt and yum/dnf — servers will be one or the other, know both
8
All issues are resolved. Write memory/linux-basics.md — the 5 command categories a DevOps engineer must own.
I'm writing memory/linux-basics.md for my DevOps agent. Generate the full file with these 5 sections: 1. Filesystem hierarchy — the purpose of /etc, /var, /tmp, /proc, /home, /usr/bin with a concrete DevOps example for each (what you look for in that directory during an incident) 2. Permission model — how to read rwxr-xr-x, numeric notation 755/644/600/700, when to use each, and why 777 is a red flag with a real security example 3. User and group management — the commands to create, modify, list, and delete users; what /etc/passwd and /etc/shadow contain and why they're split 4. Process and service management — ps flags, top key columns, kill vs kill -9, systemctl status/start/stop/restart/enable — with the systemd service lifecycle explained 5. Package management — apt and yum/dnf patterns side by side with the commands I'd run during an incident Write it for my agent to consume, not as notes for me. Structured sections, real command examples.
Save the full output into memory/linux-basics.md in your repo
