AWS Architecture Practice — Extend Your Day 6 Setup
How do you harden and extend a basic 3-tier AWS architecture with a static IP, secure outbound internet access, VPN access, and storage — without ever exposing more than necessary to the public internet?
The architecture you built on Day 6 works — but it has gaps. The public EC2 IP changes every restart. Private instances have no outbound internet. There is no way for a teammate to securely access the private subnet without opening SSH to the world. There is no object storage. Today you close all four gaps by adding Elastic IP, NAT Gateway, OpenVPN, and S3 with bucket policies — the standard additions made to every production VPC before it is handed to a development team.
⚠ Required tags — every resource you create must have these two tags
The AWS account is configured with a tag enforcement policy. If you create any resource (EC2, NAT Gateway, Elastic IP, S3 bucket, security group) without these two tags, the request will be denied with a "not authorized" error — even though you have the correct permissions. This is not a permissions bug. Add the tags first.
Tag 1 Key: Owner Value: your name — e.g. duy_nk
Tag 2 Key: Email Value: your email — e.g. duy_nk@gmail.com
# Add tags when launching EC2 via CLI:
aws ec2 run-instances ... \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Owner,Value=your_name},{Key=Email,Value=your@email.com}]'

# Add tags to an existing resource (e.g. Elastic IP, NAT Gateway):
aws ec2 create-tags --resources <resource-id> \
  --tags Key=Owner,Value=your_name Key=Email,Value=your@email.com
In the AWS console: every "Launch instance" or "Create" form has a Tags section at the bottom. Add both tags before clicking Create. This applies to EC2, NAT Gateway, Elastic IP, S3 bucket, and security group.

Build steps — complete in order
Each step is independent but order matters — NAT Gateway must exist before you update the private route tables. Elastic IP must be allocated before OpenVPN is launched. Click each step when complete.
1
Elastic IP — allocate and associate to EC2 Public
Allocate an Elastic IP from the AWS pool and associate it with your EC2 Public instance. After this, stopping and starting EC2 Public will not change its public IP address.
# Allocate an Elastic IP
aws ec2 allocate-address --domain vpc

# Associate it with your EC2 Public instance
aws ec2 associate-address \
  --instance-id i-xxxxxxxxxxxxxxxxx \
  --allocation-id eipalloc-xxxxxxxxxxxxxxxxx

# Verify
aws ec2 describe-addresses --query 'Addresses[*].[PublicIp,InstanceId]' --output table
Cost note: An Elastic IP that is NOT associated with a running instance is billed (~$0.005/hr). Always associate it immediately or release it when done.
Verify: stop EC2 Public, start it again — the public IP in the console must remain the same.
2
NAT Gateway — create in public subnet, update private route tables
Create a NAT Gateway in the public subnet (it needs its own Elastic IP). Then update the route tables for the private and database subnets so that outbound internet traffic routes through the NAT Gateway — not the Internet Gateway.
# Allocate an Elastic IP for the NAT Gateway
aws ec2 allocate-address --domain vpc

# Create NAT Gateway in the public subnet
aws ec2 create-nat-gateway \
  --subnet-id subnet-xxxxxxxxxxxxxxxxx \
  --allocation-id eipalloc-xxxxxxxxxxxxxxxxx

# Wait for NAT Gateway to become available (~60 seconds)
aws ec2 describe-nat-gateways --query 'NatGateways[*].[NatGatewayId,State]'

# Add route to private route table
aws ec2 create-route \
  --route-table-id rtb-xxxxxxxxxxxxxxxxx \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-xxxxxxxxxxxxxxxxx

# Repeat for database route table
Cost note: NAT Gateway costs ~$0.045/hr plus $0.045 per GB of data processed. Delete it when you finish the lab.
Verify: SSH into EC2 Private, run curl https://google.com — it must succeed. If it hangs, the route table is not updated.
3
OpenVPN EC2 — launch from Marketplace AMI, configure security group

This video demonstrates how to set up OpenVPN on an EC2 instance. The architecture shown in the video is different from Day 6 Extended. Watch it to understand the OpenVPN setup process — then follow the Day 6 Extended architecture (public subnet, your existing VPC 10.11.0.0/16) when you build your own.
Watch on Google Drive Jump to 1:13:00 – 1:50:00 — OpenVPN EC2 setup walkthrough
The video uses a different VPC and subnet layout. Do not copy the IP ranges or subnet structure from the video — use your Day 6 VPC (10.11.0.0/16) and place OpenVPN EC2 in your existing public subnet.
Launch an EC2 instance from the OpenVPN Access Server AMI (AWS Marketplace, free for 2 concurrent connections). Place it in the public subnet. It needs its own Elastic IP so the client config stays valid after restarts.
# Find the OpenVPN Access Server AMI in your region
aws ec2 describe-images \
  --owners aws-marketplace \
  --filters "Name=name,Values=*OpenVPN Access Server*" \
  --query 'Images[*].[ImageId,Name,CreationDate]' \
  --output table | head -20

# Security group rules needed for OpenVPN EC2:
# Inbound: UDP 1194 (OpenVPN), TCP 943 (admin UI), TCP 443 (client UI), TCP 22 (SSH setup only)
# Outbound: all (default)

# After launch — SSH in with the ec2-user key to complete setup:
ssh -i key.pem openvpnas@<openvpn-public-ip>
# Follow the setup wizard — accept defaults
# Set the admin password:
sudo passwd openvpn

# Download client config from:
# https://<openvpn-public-ip>/ (use your Elastic IP)
Use the Elastic IP for the OpenVPN instance — the .ovpn client config contains the server IP. If it changes after a restart, all clients must re-download the config.
Verify: install the OpenVPN client, import the .ovpn file, connect. Once connected, try SSH into EC2 Private using its private IP (e.g. 10.11.11.x) directly — no jump host needed.
4
S3 Public Bucket — static asset hosting with bucket policy
Create an S3 bucket for static assets (images, frontend files). Disable "Block all public access" for this bucket only, then attach a bucket policy that allows anyone to read objects. Upload a test file and access it by URL.
# Create the bucket (name must be globally unique)
aws s3 mb s3://proops2026-public-<yourname>

# Disable block public access for this bucket
aws s3api put-public-access-block \
  --bucket proops2026-public-<yourname> \
  --public-access-block-configuration "BlockPublicAcls=false,IgnorePublicAcls=false,BlockPublicPolicy=false,RestrictPublicBuckets=false"

# Attach public read bucket policy
aws s3api put-bucket-policy \
  --bucket proops2026-public-<yourname> \
  --policy '{
    "Version":"2012-10-17",
    "Statement":[{
      "Sid":"PublicRead",
      "Effect":"Allow",
      "Principal":"*",
      "Action":"s3:GetObject",
      "Resource":"arn:aws:s3:::proops2026-public-<yourname>/*"
    }]
  }'

# Upload a test file
echo "hello from s3" > test.txt
aws s3 cp test.txt s3://proops2026-public-<yourname>/

# Access via URL (no credentials needed):
# https://proops2026-public-<yourname>.s3.ap-southeast-1.amazonaws.com/test.txt
Verify: open the URL in your browser with no AWS credentials — the file must load. This is how static websites, CDN origins, and public asset buckets work.
5
S3 Private Bucket — deny public access, allow from VPC CIDR only
Create a second S3 bucket for application data and logs. Keep all public access blocked. Attach a bucket policy that allows access only from requests originating within the VPC CIDR 10.11.0.0/16. Test that EC2 Private can access it but your laptop (outside the VPC) cannot.
# Create the private bucket
aws s3 mb s3://proops2026-private-<yourname>

# Block all public access (this is the default — confirm it is on)
aws s3api put-public-access-block \
  --bucket proops2026-private-<yourname> \
  --public-access-block-configuration "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

# Attach VPC-only bucket policy
aws s3api put-bucket-policy \
  --bucket proops2026-private-<yourname> \
  --policy '{
    "Version":"2012-10-17",
    "Statement":[{
      "Sid":"VPCOnly",
      "Effect":"Deny",
      "Principal":"*",
      "Action":"s3:*",
      "Resource":["arn:aws:s3:::proops2026-private-<yourname>","arn:aws:s3:::proops2026-private-<yourname>/*"],
      "Condition":{"NotIpAddress":{"aws:SourceIp":"10.11.0.0/16"}}
    }]
  }'

# Upload a test file from your laptop
echo "private data" > private.txt
aws s3 cp private.txt s3://proops2026-private-<yourname>/

# From EC2 Private (SSH in) — this must SUCCEED:
aws s3 ls s3://proops2026-private-<yourname> --region ap-southeast-1

# From your laptop without VPN — this must return Access Denied:
aws s3 ls s3://proops2026-private-<yourname>
The aws s3 cp from your laptop to upload the initial file will succeed because the upload request originates from your IP, not the VPC. The policy blocks reads from outside the VPC. Test with aws s3 ls, not cp.
Verify: connect to VPN, then run aws s3 ls s3://proops2026-private-yourname from your laptop. With VPN on, traffic originates from the VPC CIDR — it should succeed. Disconnect VPN — same command returns Access Denied.
