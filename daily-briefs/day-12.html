Kubernetes Basics: Pod, Deployment, Service
What are the three foundational Kubernetes objects every workload uses — and how do they connect to produce a running, accessible, self-healing application?
Kubernetes is the operating system of cloud-native infrastructure. Pod, Deployment, and Service are the three objects in virtually every K8s workload — everything else you will learn (Ingress, HPA, ConfigMap, Secret) is built on top of them. A DevOps engineer who cannot write a Deployment manifest by hand, read what kubectl shows, and debug a Pod that will not start is blocked on every modern infrastructure task. Yesterday Swarm showed you WHY orchestration exists. Today you learn the three nouns that express it.

Concept — the three objects and how they connect
Every workload you deploy to Kubernetes uses these three objects. Read this before you run a single kubectl command — the mental model is what makes the commands make sense.
📦
1. Pod
The runtime unit
A Pod is one or more containers that share a network namespace and storage. It is the smallest thing K8s schedules onto a node. You almost never create Pods directly — if a Pod dies, nothing replaces it. Pods are a building block that a Deployment manages on your behalf.
🔄
2. Deployment
The desired-state manager
A Deployment says: "I want 3 replicas of this Pod template running, always." It creates a ReplicaSet that creates Pods. If a Pod dies, the ReplicaSet replaces it. If you change the image tag, the Deployment performs a rolling update — replacing Pods in controlled batches, keeping old ones alive until new ones pass health checks.
🌐
3. Service
The stable network endpoint
Pod IPs change on every restart. A Service gives you a stable DNS name and virtual IP that always routes to the live Pods behind it. The connection between a Service and its Pods is a label selector — the Service forwards traffic to every Pod carrying the matching labels. This is how Pods keep finding each other during a rolling update.
How they connect
Deployment (replicas: 3, selector: app=web) ↓ creates ReplicaSet (maintains 3 Pods matching the template) ↓ creates Pod app=web Pod app=web Pod app=web ↑ ↑ ↑ ─────── traffic routed by ────── ↑ Service (selector: app=web) ↑ Client hits stable DNS: http://web/
The four commands you will run today. Any time a Pod is not healthy, run them in this order: kubectl get pods → kubectl describe pod [name] → kubectl logs [name] → kubectl get events --sort-by='.lastTimestamp'. Each one reveals something the others do not. This sequence is muscle memory for every K8s operator.
Discovery chain — 8 questions
1
You apply a Deployment manifest and run kubectl get pods. Two Pods show CrashLoopBackOff. What does this status mean, and what is the first command you run to find out why the container is crashing — before you even look at the Deployment definition?
Explain CrashLoopBackOff in Kubernetes to me as if I have never seen it before. Cover: (1) what the status literally means — what loop is looping, what backoff is being applied, (2) the most common root causes in order of frequency (app exits on startup, missing env var, wrong image, OOMKilled), (3) the kubectl command I run FIRST to see why the container is crashing — not kubectl describe but the one that actually shows application output, (4) what the three flags for that command do: --previous, --tail, --follow, and when each one matters. Keep it short and specific — I want a diagnostic playbook, not a tutorial.
kubectl logs [pod-name] is the first command. Use --previous if the container has already restarted — you want the logs from the crashed instance, not the new one that is about to crash.
2
The logs show the app cannot find the environment variable DATABASE_URL. You look at the Deployment manifest — where do environment variables live, and what is the relationship between the Pod spec and the container spec inside it?
Show me a minimal Deployment YAML for a generic web app. Annotate where each piece lives: (1) the Deployment spec (replicas, selector, strategy), (2) the Pod template inside it — spec.template.spec, (3) the container array inside the Pod spec — spec.template.spec.containers, (4) where environment variables are defined — at the container level, not the Pod level. Show me two ways to define env vars: (a) inline value, (b) from a ConfigMap or Secret using valueFrom. Also explain why the selector.matchLabels in the Deployment MUST match template.metadata.labels — what breaks if they do not.
Every container in the Pod has its own env: list. The Pod spec shares network namespace and volumes; the container spec owns image, env, ports, probes. Use kubectl explain deployment.spec.template.spec.containers to browse all the container fields from the command line — this is the single best learning tool for manifest syntax.
3
You add the env var. The Pod starts. But kubectl get pods shows 0/1 READY. The container is running but the readiness check is failing. What is the difference between a liveness probe and a readiness probe, what does "not ready" mean for traffic, and how do you read a failing probe?
Explain liveness vs readiness probes in Kubernetes using one concrete scenario: a web app that takes 20 seconds to warm up at startup, then connects to a database that can go down intermittently. For each probe type: (1) what question is the probe answering, (2) what does Kubernetes do when the probe fails — restart the container, or stop sending traffic, or both, (3) what should the probe endpoint check — the process, a simple HTTP 200, or the database connection, (4) what does 0/1 READY mean in kubectl get pods output, and how does it affect Service routing. Finally, show me the YAML for both probes on a container using httpGet, with sensible initialDelaySeconds and periodSeconds values for the scenario above.
Liveness failing = container restarted. Readiness failing = Pod removed from Service endpoints (no traffic), but container keeps running. If your readiness probe checks the database, a DB blip causes ALL your Pods to go 0/1 at once — this is a classic production outage pattern.
Read the describe output carefully: kubectl describe pod [name] shows the last probe failure reason under the Events section. That is where "connection refused", "404", or "timeout" appears.
4
The readiness probe passes. The Pod is 1/1 READY. You try to reach the app from outside the cluster — not accessible. The Service is type ClusterIP. What do the four Service types give you access to, and which one do you need?
Give me a comparison of the four Kubernetes Service types: ClusterIP, NodePort, LoadBalancer, ExternalName. For each one: (1) what problem it solves, (2) who can reach the Service (inside the cluster only, from the node IP, from the internet), (3) what IP/port you use to connect, (4) when you would pick it over the others. Then show me a minimal Service YAML for a NodePort type that targets Pods with label app=web on container port 8080. Explain the three ports in the YAML — port, targetPort, nodePort — and what happens if I omit nodePort.
ClusterIP = inside cluster only (default). NodePort = reach via any node IP on a high port 30000–32767. LoadBalancer = cloud provider provisions an external load balancer (minikube simulates this with minikube tunnel). ExternalName = DNS alias to an external service, no proxying.
5
You change the Service to NodePort. You can reach the app. You notice traffic hits different Pods on different requests. How does the Service know which Pods belong to it — what is a label selector and how does it stay current during a rolling update?
Explain label selectors in Kubernetes. Walk me through: (1) what labels are — arbitrary key/value pairs attached to Pods and other objects, (2) how the Service's spec.selector field connects to Pods — via matchLabels, (3) what kubectl command shows the list of Endpoints currently backing a Service — what are Endpoints objects, and how are they different from the Service itself, (4) during a rolling update when new Pods appear with the same labels and old Pods are terminating, how does the Endpoints controller keep the list current, and what prevents a client from hitting a Pod that is mid-shutdown. Keep it concrete — use a Deployment named "web" with label app=web as the example.
Run kubectl get endpoints [service-name] to see the actual list of Pod IPs behind a Service. During a rolling update this list updates in real time. The terminationGracePeriodSeconds field on a Pod (default 30s) is what gives the Endpoints controller time to remove a terminating Pod before it is killed.
6
You need to update the image to a new tag. You apply the change. Kubernetes performs a rolling update. How does the RollingUpdate strategy work — what do maxUnavailable and maxSurge control — and how do you roll back if the new version is broken?
Walk me through a Kubernetes rolling update on a Deployment with replicas: 4. Assume strategy.type: RollingUpdate with default maxUnavailable and maxSurge. Answer: (1) what the defaults are for maxUnavailable and maxSurge (percentage or absolute), (2) with defaults, what is the maximum number of Pods alive at any point during the update, and the minimum number ready, (3) what happens if a new Pod fails its readiness probe mid-update — does the rollout stop, or keep trying, (4) the exact command to check rollout status in real time, (5) the exact command to roll back to the previous revision, (6) how Kubernetes tracks revisions — where are old ReplicaSets kept and how many by default. End with a one-paragraph explanation of why aggressive maxSurge (like 100%) might be bad on a resource-constrained cluster.
kubectl rollout status deployment/web watches progress. kubectl rollout undo deployment/web reverts to the previous ReplicaSet. kubectl rollout history deployment/web shows revisions. Old ReplicaSets are kept so undo works — the revisionHistoryLimit field controls how many.
7
After the rollout, kubectl get pods shows 3 running, but kubectl get deployments shows 2/3 READY. One Pod is stuck. What do describe, logs, and get events each reveal that the others do not?
Build me a kubectl diagnostic workflow for a stuck Pod. Show me the exact command for each step and explain what information it gives you that the others miss: 1. kubectl get pods — what columns matter (READY, STATUS, RESTARTS, AGE) 2. kubectl describe pod [name] — what sections to read first (Events, Conditions, Containers with State + Last State) 3. kubectl logs [name] — when to use --previous vs current 4. kubectl get events --sort-by='.lastTimestamp' — why cluster-wide events sometimes reveal problems Pod events do not (scheduling failures, image pull errors from the kubelet) 5. kubectl exec -it [name] -- sh — when you reach for this as a last resort For each command, give me one concrete failure signal that command reveals that nothing else does.
describe shows the last probe error and scheduling decisions. logs shows what the app actually said before dying. get events shows cluster-wide things like "0/3 nodes are available: insufficient memory" that never appear in a Pod's own events.
Run this sequence against a Pod you deliberately broke today (wrong image tag, bad env var, probe pointing at the wrong port). Seeing each command's unique output in a real failure sticks far better than reading about it.
8
Deployment is healthy. Write memory/kubernetes-core.md — the three object relationships, probe types, the kubectl diagnostic workflow, and one working manifest example per object, all in a form your agent can query fast.
Help me write memory/kubernetes-core.md. This file loads into my agent's context every time I work on Kubernetes. It must contain: 1. The three objects — one short section each: - Pod: what it is, what containers in a Pod share (network, volumes), why you do not create Pods directly - Deployment: desired state, ReplicaSet relationship, rolling update strategy (maxUnavailable, maxSurge), rollback command - Service: the four types (ClusterIP/NodePort/LoadBalancer/ExternalName), label selectors as the connection mechanism, what happens during a rolling update 2. Liveness vs readiness probes — two sentences each plus the YAML fields (httpGet, initialDelaySeconds, periodSeconds) 3. The kubectl diagnostic workflow — 4-step sequence with one line per step on what it uniquely reveals (get pods → describe → logs → get events) 4. One minimal working manifest per object: Pod, Deployment, Service — each under 20 lines, all three labeled consistently so they connect 5. A table: "If I see X, run Y" — common statuses (CrashLoopBackOff, ImagePullBackOff, 0/1 READY, Pending, Error) mapped to the first diagnostic command Format for an AI agent, not a human reader: tables and short bullets beat paragraphs.
The three working manifests are the highest-value part — your agent can reference them directly when it writes new ones. Keep them labeled app=[something] consistently so a reader immediately sees how the Service selector, Deployment selector, and Pod labels all match.
