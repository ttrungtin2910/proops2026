Docker Compose: Volumes, Networking, Registry
How do you run a multi-service application as a single coordinated system — and how do containers share data, talk to each other, and get their images to a registry?
Real applications are never a single container. A service, a database, and a cache each run in their own container and must communicate reliably. Docker Compose is the tool that defines this entire system as code — so any engineer can bring up the full stack with one command, on any machine, every time. Without Compose, every multi-container setup is a manual sequence that no one can reproduce. Without volumes, every database restart loses all data. Without the registry, your CI/CD pipeline has no image to pull. Today you learn all three — then apply them to your own mock project stack.
Group work continues. You and your partner work in the same GitHub org today. The docker-compose.yml you write lives in your docs repo (or a dedicated infra repo — your call, decide together before starting). At the end of the day, both partners should be able to run docker compose up from their own machine and get the same result.

Discovery chain — 8 questions, one working stack
Q1–Q7 teach the concept using a generic example. Work through them fully before touching your project. Q8 is where you apply everything to your own stack. Do not skip ahead — the concepts must be understood before the application makes sense.
1
You have two services — an API and a PostgreSQL database. Running them with separate docker run commands in the right order works on your machine. Why does this break for everyone else, and what does Docker Compose solve?
I have two containers: an API service and a PostgreSQL database. I start them with two separate docker run commands, manually setting the right ports, environment variables, and startup order. Explain: (1) what are the three specific problems with this manual approach — reproducibility, startup order, and network configuration, (2) what does a docker-compose.yml file solve for each of those three problems, (3) what is the structure of a docker-compose.yml — what do the services, networks, and volumes top-level keys each control? Show me a minimal working example with just these two services.
After Claude explains, read the minimal example carefully. Every field has a reason — understand what services, networks, and volumes each mean before moving on.
2
You add depends_on: db to your API service. The API still crashes on first run because the database isn't ready yet. What does depends_on actually check — and what is the real solution?
I added depends_on: db to my API service in docker-compose.yml but the API still crashes on startup because the database isn't ready to accept connections. Explain: (1) what does depends_on actually wait for — container started vs process ready — and why that distinction matters, (2) what is a Docker healthcheck — how do I write a healthcheck for postgres:16-alpine that tests whether it can accept connections using pg_isready, (3) how does depends_on with condition: service_healthy differ from depends_on with no condition, (4) write the complete healthcheck block for a PostgreSQL service and the matching depends_on condition block for the API service that depends on it.
depends_on without a condition is almost always wrong for databases. The healthcheck is what turns it from "container running" to "database ready."
3
You stop the stack with docker compose down and restart it. All data in the database is gone. Why — and how do named volumes fix this?
I stopped my Compose stack with docker compose down and restarted it. All rows in my PostgreSQL database are gone. Explain: (1) why is the container filesystem ephemeral — what happens to data when a container is removed vs stopped, (2) what is a Docker named volume — where does Docker actually store the data on the host, (3) what is the difference between a named volume and a bind mount — give me the one-line rule for when to use each, (4) write the volumes section of a docker-compose.yml that mounts a named volume to the PostgreSQL data directory /var/lib/postgresql/data. Also show what docker compose down -v does and when you would use it intentionally.
One-line rule to remember: named volume = Docker manages it, for data you want to persist (databases). Bind mount = you manage it, for files you need to read/edit from the host (logs, dev code).
4
Both containers are running, but the API can't connect to the database with hostname db. You try localhost — it also fails. What is Docker Compose's network model and how do you verify it works?
My API container tries to connect to my database container using the hostname localhost — connection refused. I try the Compose service name db — still fails. Explain: (1) what network does Docker Compose create automatically and what is its name format, (2) how do containers in the same Compose network resolve each other by service name — which DNS mechanism makes this work, (3) why does localhost fail — what does localhost refer to inside a container vs on the host machine, (4) what is the correct DATABASE_URL connection string for a service named db on port 5432 with a database named mydb. Finally: how do I verify connectivity from inside the API container using docker exec and curl — show me the exact commands.
Run the verification — do not just read it. Open a terminal, run docker exec into the API container, and curl the db service by its Compose name. Watch the DNS resolution work.
This is the key realization of today: containers in the same Compose network reach each other by service name, not IP, not localhost. Do it with your hands — not just your eyes.
5
Your stack works but the docker-compose.yml has POSTGRES_PASSWORD: mysecret hardcoded. How do you handle secrets and configuration safely?
My docker-compose.yml has POSTGRES_PASSWORD: mysecret hardcoded. I cannot commit this to the repo. Explain: (1) how does the env_file directive work — how does Compose read environment variables from a .env file into a service, (2) what is the difference between env_file and the environment key in a service definition — when would you use each, (3) what is the .env.example pattern — what goes in .env (gitignored) vs .env.example (committed), (4) rewrite a postgres service definition to use env_file instead of hardcoded values. Show the final docker-compose.yml snippet, the .env file content, and the .env.example file content. Also show the .gitignore entry needed.
The .env file is never committed. The .env.example shows every key with an empty or placeholder value. Every teammate clones the repo, copies .env.example to .env, and fills in real values.
6
Your image works locally. Now push it to Docker Hub so your teammate can pull it on their machine without having the source code. What is the registry workflow?
My service Docker image builds correctly on my machine. I need to push it to Docker Hub so my teammate can pull it directly without needing my source code. Walk me through: (1) docker login — how to authenticate to Docker Hub from the CLI, (2) docker tag — what is the full image name format (username/repo:tag) and why does the tag matter, (3) docker push — what happens layer by layer when you push, (4) docker pull — how does my teammate pull and run the exact same image, (5) what is the difference between Docker Hub, AWS ECR, and a private self-hosted registry — when would a team choose each? Give me the complete command sequence: build → tag → push → pull, with real placeholder values filled in.
Always tag with a meaningful version — e.g. username/myservice:day09. The :latest tag is ambiguous and causes silent bugs in CI/CD pipelines.
7
Your teammate pulls the image on their M1/M2 Mac and gets exec format error. What is the multi-platform build problem and how do you fix it?
My teammate pulled my Docker image on their Apple M1 Mac and got "exec format error". I built the image on an Intel x86_64 machine. Explain: (1) why does CPU architecture matter for Docker images — what is the difference between linux/amd64 and linux/arm64, (2) what is docker buildx — how is it different from docker build, (3) what does the --platform flag do when building, (4) what is a multi-platform image manifest and how does Docker Hub automatically serve the right image per platform, (5) give me the exact docker buildx build command to build and push an image for both linux/amd64 and linux/arm64 at the same time.
For this program, linux/amd64 is all you need — that is what EC2 and CI runners use. Multi-platform only becomes important when teammates are on M-series Macs or you are deploying to ARM-based servers.
8
Apply it. Write the docker-compose.yml for your actual mock project — all your services, all your databases, running together with one command.
I've learned Docker Compose concepts (Q1–Q7). Now I need to write a docker-compose.yml for my real project. My project has these services: [list your services from your DOP]. For each service, here is what it needs: [describe: which other services it depends on, which database it uses, which port it listens on, what env vars it needs]. Write a complete docker-compose.yml that: (1) defines all services with build context pointing to each service's repository, (2) uses a named volume for each database, (3) puts all services on a custom named network, (4) uses env_file for every service — show me the matching .env.example too, (5) adds a healthcheck for each database with depends_on condition: service_healthy for services that need the DB ready, (6) only exposes the gateway on a host port — all other services and databases are internal only. After writing it, tell me: what is the first thing I should check to verify this file is correct before running it?
Read the output carefully before running it. Verify: service names match what your application code uses as hostnames. Port numbers match what each service listens on. Env var names match what the service code reads.
If your project docs define infrastructure requirements (gateway routing, port rules, network layout), check that your Compose file matches. If anything differs from your docs, update the docs first — then the file.
