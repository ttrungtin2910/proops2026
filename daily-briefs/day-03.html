Git Basics + Networking Basics
How does code move from a developer's laptop to a server — and what does the network infrastructure look like that carries that traffic?
Git is the source of truth for every artifact a DevOps engineer manages — not just application code, but Dockerfiles, Kubernetes manifests, Terraform configs, and CI/CD pipeline definitions. Networking is the substrate everything runs on: a deployment that "works on my machine" fails in production almost always because of a network or DNS issue. You cannot debug a broken pipeline or a failed deployment without understanding both.

1
A developer says "I pushed but the pipeline didn't trigger." What git commands tell you the current branch state, recent commits, and whether the push actually reached the remote?
I'm a DevOps engineer. A developer says they pushed code but the CI/CD pipeline didn't trigger. Walk me through the exact git commands I'd run to check: which branch they're on, the last 5 commits, whether the push reached the remote, and the current state of the working tree. Show me the commands AND explain what each output tells me.
Key concept: git status, git log, git remote — reading the state of a repo
2
The push was to feature/login but the pipeline triggers on main. What is the git concept behind branches, and how do you merge feature→main without losing history?
The developer pushed to feature/login but the pipeline only triggers on main. Explain: (1) why branches exist in git and what they actually are under the hood, (2) the exact commands to merge feature/login into main — both merge and rebase approaches, (3) when to use merge vs rebase and why rebase rewrites history. Show me the commands I'd actually type.
Key concept: branches as pointers, merge vs rebase, fast-forward
3
The merge worked, but the new version is broken in production. You need to roll back. What are the two strategies (revert vs reset), and which one is safe on a shared branch?
I merged a feature branch into main and it broke production. I need to roll back. Explain the two strategies — git revert and git reset — with the exact commands for each. Show me what each one does to the git history (draw an ASCII diagram). Then tell me the safety rule: which is safe on a shared branch like main, and why the other one is dangerous when other people have already pulled.
Key concept: revert = new commit that undoes. reset = rewrite history. Shared branch = revert only.
4
After rolling back, you need to find what line of code caused the regression. How do you use git diff and git log to pinpoint the breaking commit?
The production rollback worked. Now I need to find the exact commit that introduced the bug. Show me: (1) git log with useful flags to narrow down the suspect commits, (2) git diff to compare two commits and see exactly what changed, (3) git bisect — what it is and when it's worth using over manual log reading. Give me the actual commands with realistic examples.
Key concept: git as a debugging tool — log, diff, bisect, blame
↓
The code is fixed and deployed. But users still can't reach the app. The problem is now the network — not the code.
5
What is the complete sequence — DNS, TCP, TLS, HTTP — that must succeed for a browser to reach the app? Where do most outages live in that chain?
Walk me through the complete sequence when a browser requests https://app.example.com — from DNS resolution to the final HTTP response. Show every layer: (1) DNS resolution chain (browser cache → resolver → authoritative), (2) TCP 3-way handshake, (3) TLS handshake and what it establishes, (4) HTTP request/response. For each layer, tell me the most common failure mode that causes outages in production.
Key concept: each layer can fail independently — you diagnose from the bottom up
6
You can ping the server IP but curl https://app.example.com times out on port 443. What does this tell you, and what three CLI tools do you run next?
I can ping the server IP successfully, but curl https://app.example.com hangs and times out on port 443. Diagnose this: (1) what does "ping works but curl doesn't" tell me about which network layer the problem is at, (2) show me the 3 CLI tools I run next to narrow it down — dig (DNS), nc or telnet (port connectivity), traceroute (path). Give me the exact commands with example output and how to read each result. Also show me the 10 port numbers every DevOps engineer must know.
Key concept: layer-by-layer diagnosis — ping = ICMP, curl = TCP+TLS+HTTP, each tests different things
7
You opened port 443 in the security group, but HTTPS fails with a certificate error. What is a TLS certificate, what does this error mean, and how do you inspect the cert?
I opened port 443 and the TCP connection succeeds, but the browser shows a certificate error. Explain: (1) what a TLS certificate is and why HTTPS needs one, (2) the 3 most common certificate errors and what each means (expired, wrong hostname, self-signed), (3) the exact openssl command to inspect the certificate the server is presenting — show me the command and how to read the important fields in the output. Also explain what "stateful" means for firewall/security group rules — inbound vs outbound.
Key concept: TLS is the last gate before HTTP works. openssl s_client is your debugging tool.
8
Write your two memory files. What are the five git commands a DevOps engineer runs every day, and what are the five networking concepts that explain 90% of connectivity failures?
I'm writing two memory files for my DevOps agent. Generate both in a single response. FILE 1: memory/git.md Include: (1) the six core commands (clone, add, commit, push, pull, branch) with real flag examples, (2) branch operations — create, switch, merge, plus when rebase vs merge, (3) conflict resolution step-by-step with conflict marker examples, (4) revert vs reset with the shared branch safety rule, (5) a "daily DevOps git workflow" section — how git applies to Dockerfiles and Terraform, not just app code. FILE 2: memory/networking-basics.md Include: (1) DNS resolution chain, (2) port table — at least 10 ports with numbers, (3) firewall/security group rules — inbound vs outbound, stateful explained, (4) TLS handshake + how to debug with openssl, (5) connectivity debug checklist — the 4 commands to run in order when a service is unreachable.
This prompt generates both artifact files. Save each output into the correct path in your repo.
