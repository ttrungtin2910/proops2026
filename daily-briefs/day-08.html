The agile hierarchy — from vision to daily task
EPIC
A feature area. Maps to a DOP service. e.g., "User Auth", "Task Management"
USER STORY
"As a [user], I want to [action] so that [value]." Maps to a DOP acceptance criterion.
TASK
Concrete work unit. Max 1 day. e.g., "Write POST /tasks endpoint + test". Assigned to one person.
SPRINT
Fixed time-box of tasks (for this program: 1 week). Produces a working increment.
For a 2-person team: one person is Product Owner (owns the DOP — decides what goes into the sprint). The other is Scrum Master (owns the process — keeps the sprint moving). Both are Developers (do the work). On Day 8, the Product Owner drives the DOP/IRD design. The Scrum Master builds the Sprint 1 backlog and assigns tasks. Roles can rotate each sprint.

The document system — what DOP and IRD mean for a software project
Two documents. Two questions. One system.
DOP — Definition of Product
Answers: What are we building and why?
• What problem does this system solve?
• Who uses it? What do they do?
• What services exist and what does each own?
• What does success look like (acceptance criteria)?
Usually 1 DOP for the whole product. Add a new DOP only when the product scope changes significantly.
IRD — Implementation Requirements
Answers: How must THIS service/topic be built?
• What is the API contract for this service?
• What does this service's data model look like?
• What are the rules specific to this service?
• What non-functional requirements apply?
One IRD per service or topic. You will have many IRDs. Each one is small and focused.
Common mistake: writing one big IRD for the whole system
Wrong
DOP-001.md (product)
IRD-001.md (everything)
IRD-001 grows to 400 lines. Claude loads the whole file for every task. Section for user-service is buried next to Docker config next to task-service API contract. No one reads it.
Correct
DOP-001.md (product)
IRD-001.md (user-service)
IRD-002.md (task-service)
IRD-003.md (gateway + infra)
IRD-004.md (CI/CD) ← Day 11
IRD-005.md (monitoring) ← Day 15
Each IRD is 60–120 lines. Claude loads DOP + the one relevant IRD. Clear, focused, growable.

The rule: Write the IRD before you write the code. If you start coding without an IRD, you are guessing. If an IRD grows past ~150 lines, split it into two.
How Claude Code uses your documents: Before every implementation task, Claude reads your DOP and the relevant IRD. DOP tells Claude what to build. IRD tells Claude exactly how to build it for this service. If a requirement is missing, Claude will guess — and every wrong guess costs debugging time. The documents are the source of truth, not the chat history. Update the IRD, then implement.

End-to-end example — Task Management System
This is a complete worked example. Read it carefully before you write your own. Notice: the DOP defines what with acceptance criteria. The IRD defines how with enough precision that Claude can implement without guessing. The architecture diagram shows how services connect.
DOP-001 Example Task Management System
Problem statement
Teams tracking tasks in chat tools (Slack, Discord) lose context — tasks get buried in messages, ownership is unclear, and there is no way to see what is blocked. This system gives teams a central place to create, assign, and track tasks via a REST API.

Users and use cases
• Team member — create a task, assign it to a person, mark it done, add a comment
• Team lead — view all tasks by status, filter by assignee, see overdue tasks
• System (automated) — receive task events via queue for notification processing

Services
• api-gateway — single entry point, routes requests to services, handles auth token validation
• task-service — owns task CRUD, task status transitions, task assignment
• user-service — owns user accounts, login, JWT token issuance
• notification-service — consumes task events from queue, sends email/webhook notifications
• PostgreSQL — persistent storage for tasks and users (separate databases per service)
• Redis — session cache and queue (optional for Day 8, required by Day 12)

Acceptance criteria
• A user can register, login, and receive a JWT token
• A logged-in user can create a task with title, description, assignee, due date
• A task moves through states: TODO → IN_PROGRESS → DONE → CANCELLED
• A team lead can list all tasks filtered by status and assignee
• When a task is assigned or completed, an event is published to the queue
• All API endpoints return JSON and use standard HTTP status codes
IRD-001 Example Task Management System — Implementation Requirements
Technology stack
• api-gateway: Node.js (Express) — lightweight proxy + auth middleware
• task-service: Node.js (Express) + Prisma ORM
• user-service: Node.js (Express) + bcrypt + JWT
• notification-service: Node.js (worker process) — no HTTP server
• Database: PostgreSQL 15 (one schema per service, same instance for dev)
• Queue: Redis Streams (simple, no additional infra)

API contract — task-service
POST   /tasks              → create task      → 201 { id, title, status, assigneeId }
GET    /tasks              → list tasks       → 200 [{ id, title, status, assigneeId }]
GET    /tasks/:id          → get task         → 200 { id, title, description, status, ... }
PATCH  /tasks/:id          → update task      → 200 { id, status, ... }
DELETE /tasks/:id          → delete task      → 204
Data model — task-service
Task {
  id          UUID primary key
  title       VARCHAR(255) not null
  description TEXT
  status      ENUM(TODO, IN_PROGRESS, DONE, CANCELLED) default TODO
  assigneeId  UUID references user(id)
  createdBy   UUID references user(id)
  dueDate     TIMESTAMP nullable
  createdAt   TIMESTAMP default now()
  updatedAt   TIMESTAMP
}
Service communication rules
• All inter-service calls go through api-gateway — services do NOT call each other directly
• Events published to queue on: task.created, task.assigned, task.completed
• Event schema: { eventType, taskId, userId, timestamp }
• notification-service is consumer-only — it never exposes HTTP endpoints

Non-functional requirements
• Each service runs in its own Docker container
• Configuration via environment variables only — no hardcoded values
• Health check endpoint: GET /health → 200 { status: "ok" } on every HTTP service
• Database migrations managed by Prisma — no manual SQL

Write your two sentences yourself first. Then paste into Claude. Do not ask Claude to write the problem statement for you.
2
What are the core user actions — and which service owns each one? List them before asking Claude to validate.
My application has these user types: [list]. For each user type, these are the actions they perform: [list actions per user]. Based on this, I think I need these services: [list your services]. (2) find any action that has no clear service owner, (3) check if my service boundaries follow the principle that each service owns its own data and does not share a database table with another service.

List your services and their responsibilities yourself first. Claude validates — it does not design for you.
3
For each core user action, trace the full CRUD flow end to end — from the user's request to the database write and back.
I'm tracing the full request flow for my application. Walk me through a complete example for each of these core actions: [list your 3–5 most important actions]. For each action: (1) which HTTP method and endpoint does the client call? (2) which service handles the request? (3) what database operation happens (INSERT, SELECT, UPDATE, DELETE)? (4) what is returned to the client and what HTTP status code? (5) if another service needs to be notified (e.g. for events), how does that happen — synchronous HTTP call or async queue message? I want to see the full flow for each action in a table or step-by-step format.
If you can't trace the flow, the system is not defined. Every gap here becomes a bug later.
4
Design the data model for each service. What tables, what columns, what relationships — before any code is written.
I'm designing the data model for my microservice application. For each service I have: [list your services]. Help me design the data model for each: (1) what tables does each service own? (2) for each table: what are the columns, data types, constraints (not null, unique, default), and primary key? (3) what are the relationships — foreign keys within the service, and how do cross-service references work (hint: store only the ID, not a foreign key, when referencing another service's data), (4) what indexes are needed for the queries I described in Q3? Write the data model in a format I can paste directly into my IRD.
Cross-service data: store only the ID. Never a foreign key to another service's table. Services must be independently deployable.
5
Define the API contract for each service — method, path, request body, response body, and error codes.
I need to define the complete API contract for my microservice application. For each service, generate a full API contract table with: HTTP method, path, description, request body (JSON schema), success response (HTTP status + JSON schema), and the two most common error responses (e.g. 400, 404, 401). Services: [list your services and their responsibilities]. Rules: (1) use REST conventions — plural nouns for resources, correct HTTP verbs, (2) all endpoints return JSON, (3) error responses must include a `message` field, (4) authenticated endpoints must note that a JWT Bearer token is required. Format as a table I can paste into my IRD.
The API contract is the agreement between services. If it changes without updating the IRD, the system breaks silently.

Sprint 1 Planning — manual backlog, before any automation
You have your DOP-001 and IRD-001. Now you run Sprint 1 Planning as a team. This session is manual — no Linear, no automation yet. You write the backlog by hand in docs/sprint-01.md. On Day 9, you start implementing it. On Day 10–11, you will automate this exact process using /create-linear-issue.
Step 1 · Epics
One Epic per DOP service. Name it after the service. e.g., "Epic: User Service", "Epic: Task Service". The Epic is the container — everything in it belongs to that service.
Step 2 · User Stories
Take each DOP acceptance criterion. Rewrite it as: "As a [user type], I want to [action] so that [value]." One story per AC. A good story fits in one sprint. If it doesn't, split it.
Step 3 · Tasks + Assign
Break each user story into concrete tasks ≤ 1 day each. Assign each task to Person A or Person B. Sprint 1 is the first service only — do not put the whole system in Sprint 1.
