Virtualization + Cloud Concepts
What are the layers of abstraction between your application and physical hardware — and why do those layers exist in the first place?
Every cloud resource you will ever provision is a virtualization product. Understanding the stack — physical server → hypervisor → VM → container runtime → container — explains why things behave the way they do: why containers start in milliseconds while VMs take minutes, why a VM crash doesn't affect its neighbors, why "the cloud" still has capacity limits. Without this model, every cloud architecture decision is guesswork. With it, you can reason about cost, performance, isolation, and failure from first principles.

Discovery Chain — 8 questions, one abstraction stack
Today's 8 questions build the mental model from the ground up — starting with why virtualization exists, through VMs and containers, up to cloud architecture decisions and cost. Copy each purple prompt, paste into Claude, read the answer carefully, then rephrase the key idea in your own words before moving on.
1
Your company runs an app on one physical server. It sits at 10% CPU most of the day but spikes to 90% at peak hours. What does virtualization solve that buying a bigger server does not?
I'm running an application on a single physical server. It uses 10% CPU most of the day but spikes to 90% during business hours. My manager says "just buy a bigger server." Explain to me: (1) why that doesn't actually solve the core problem, (2) what virtualization solves instead — specifically, what a hypervisor does and what resource isolation means, (3) why running multiple VMs on one physical host is economically and operationally better than one app per server. Use concrete numbers in the explanation.
The core insight: physical resources are fixed, but virtual allocation is flexible
2
You provision a VM on AWS — 4 vCPUs, 16GB RAM. The physical host has 32 cores and 256GB RAM with 12 VMs. What is the hypervisor actually doing, and what happens when all 12 VMs spike at once?
I just provisioned an EC2 instance with 4 vCPUs and 16GB RAM. Explain to me: (1) what the hypervisor is doing between my VM and the physical hardware — specifically the difference between Type 1 and Type 2 hypervisors and which type AWS uses, (2) what "CPU oversubscription" means — if the physical host has 32 cores and 12 VMs each claim 4 vCPUs, that's 48 vCPUs on 32 physical cores, how is that possible and what's the risk, (3) what AWS does at the infrastructure level to prevent one noisy neighbor from degrading other VMs on the same host.
Oversubscription is why cloud is cheap — and why "burstable" instances exist
3
You package your app as a Docker container (200MB) instead of a VM image (10GB). What is the technical difference in how they achieve isolation — and why is the container 50x smaller?
I need to deploy the same application as either a Docker container or a VM image. The container is 200MB. The VM image is 10GB. Explain: (1) technically how a VM achieves isolation (full OS kernel, virtual hardware) vs how a container achieves isolation (Linux namespaces and cgroups — explain what each of these does), (2) why the container is so much smaller — what is in the VM that is not in the container, (3) the startup time difference and why — what has to happen when a VM boots vs when a container starts. Give me the mental model, not just the facts.
Containers share the host kernel — that's the source of both the speed advantage and the security trade-off
4
A security auditor asks: "Is our container environment more or less isolated than our VM environment?" What is the honest answer?
A security auditor asks me whether our containerized application is more or less secure than a VM-based deployment. Give me: (1) the honest answer — with the specific technical reason why containers provide weaker isolation than VMs, (2) what "shared kernel" means as a security risk — what attack surface exists that a VM does not expose, (3) one real-world container escape vulnerability category that has been exploited in production (no need for details — just the class of attack), (4) what security controls mitigate container isolation weaknesses in production (seccomp, AppArmor, rootless containers, etc.) — one sentence each. I need to be able to answer this question in an interview and in a design review.
Honest answer: less isolated. Understanding why is what makes you credible
5
Your team is choosing between AWS EC2 (IaaS), Elastic Beanstalk (PaaS), and a SaaS product. What exactly changes about what you manage at each layer?
My team needs to choose between AWS EC2 (IaaS), AWS Elastic Beanstalk (PaaS), and a third-party SaaS tool for a new service. Explain: (1) draw out the shared responsibility stack for each — at IaaS, PaaS, and SaaS, list exactly what the provider manages and what I manage (OS patching, runtime, middleware, application code, data, network config), (2) how this choice changes the size and skills required of my operations team, (3) a concrete example: if I choose EC2 vs Beanstalk for deploying a Node.js API, what work does Beanstalk do that I would have to do myself on EC2? Keep it practical — this is a decision I'll have to make and justify.
More control = more responsibility. The right choice depends on your team's capacity
6
The CTO asks you to design for high availability. You propose multi-AZ. What is an Availability Zone — and what failure does it protect against that a single-AZ deployment does not?
My CTO asked me to design for high availability on AWS and I proposed deploying across multiple Availability Zones. Explain to me: (1) what an Availability Zone physically is — what "physically separate data center" means in practice (separate power, cooling, networking), (2) what specific failure scenarios multi-AZ protects against versus a single-AZ deployment — give concrete examples (power outage, flood, hardware failure, network partition), (3) what multi-AZ does NOT protect against — what failure would take down all AZs in a region simultaneously, (4) what the trade-off is — what does multi-AZ cost you in terms of latency, data consistency, and operational complexity? I need to be able to defend this design decision.
AZ = failure domain. Understand what it protects against AND what it doesn't
7
Finance says the cloud bill doubled last month. Nobody changed the application. What are the three most likely causes — and what monitoring would have caught each one?
Our cloud bill doubled last month and nobody made any application changes. I need to investigate and explain it to finance. Walk me through: (1) the three most common causes of unexpected cloud cost increases when code hasn't changed (data transfer costs, over-provisioned instances left running, forgotten resources like old snapshots or load balancers, traffic spikes), (2) for each cause: what AWS Cost Explorer or CloudWatch metric would have shown the spike before the invoice arrived, (3) what a basic cloud cost hygiene checklist looks like — the 5 things a DevOps engineer should check weekly to prevent surprise bills. Give me enough to run a real cost investigation.
Data egress is the most common invisible cost — know it before your first production deployment
8
Write memory/cloud-concepts.md — the five foundational cloud concepts every DevOps engineer must own.
I'm writing memory/cloud-concepts.md for my DevOps agent. Generate the full file with these 5 sections: 1. Virtualization stack — physical → hypervisor (Type 1 vs Type 2) → VM → container runtime → container. Include: what each layer does, startup time comparison, size comparison, when to use VM vs container 2. VM vs container isolation — side-by-side comparison: isolation mechanism, startup time, image size, portability, security boundary, when each is the right choice 3. IaaS / PaaS / SaaS — table of what you manage at each layer, one real AWS and one real Azure example for each, and the DevOps implication of each choice (team size, skill requirements, operational overhead) 4. Availability Zones and high availability — what an AZ is, what multi-AZ protects against, what it does not protect against (region failure), trade-offs (latency, cost, complexity) 5. Cloud cost mechanics — the three biggest sources of surprise bills (data transfer, over-provisioned instances, forgotten resources), one monitoring signal for each, and a weekly hygiene checklist Write it for my agent to consume, not as personal notes. My agent needs to be able to make architecture recommendations from this file.
