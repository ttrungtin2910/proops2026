EKS Practice — Ship Your Project from Minikube to AWS
You can deploy your project to a local cluster. Today: can you deploy the same project to a real AWS-managed Kubernetes cluster, get external traffic flowing through it, and tear it down without leaving a $200 surprise on the bill?
The skills are not new — Docker, Kubernetes, Helm, AWS basics, you already have all of them. What's new is integration under real-world constraints. EKS has more cost surface, more configuration knobs, and more opportunities to lose a day on the wrong problem than any service in AWS. Today you ship the same project you ran in minikube on Day 14–15, learn the EKS-specific moves, and — most importantly — build the muscle to create a cluster, prove it works, and delete it cleanly.
Money rule for today (read this twice)
An EKS control plane charges $0.10/hr from the moment you create it — about $2.40/day, every day, even idle. Your cluster MUST be deleted before end of day. We're not creating long-running infra today. We're learning the lifecycle create → use → delete. If you forget step 3, you discover the cost on next month's bill. Set a phone alarm right now: "Delete EKS cluster" for 17:00.
Group day — but each partner creates their own cluster
Both partners create + use + delete their own EKS cluster today. Don't share a single cluster — the lesson is the full lifecycle, and each of you needs to own that experience. Use different cluster names (e.g. project-name-huy and project-name-tuan) and both partners delete by 17:00.

Day-1 skip list — what NOT to do today
EKS is huge. Production setups have dozens of moving parts. For your first cluster, skip everything below. Each one is a valid topic for a future day, but doing all of them today is how trainees lose 6 hours and create nothing.
Skip: AWS Load Balancer Controller. Use Service type: LoadBalancer → AWS auto-provisions a Classic ELB. Free of setup pain.
Skip: IRSA (IAM Roles for Service Accounts) + OIDC provider. Use the node IAM role with AmazonEC2ContainerRegistryReadOnly attached.
Skip: Cluster Autoscaler / Karpenter. One fixed-size managed node group is enough.
Skip: Fargate. Managed node group with EC2 spot is faster and cheaper for learning.
Skip: Private endpoint. Keep the API server endpoint public so kubectl from your laptop just works.
Skip: CloudWatch Container Insights. Adds cost and complexity. kubectl logs is enough today.
Do use: eksctl with a config file. Console clicks for EKS will eat your day.
Do use: Spot instances (~70% off). Fine for ephemeral training clusters.
Discovery chain — 8 questions
1
Pre-flight check. Install eksctl, confirm your AWS identity is right, pick a region, and write down your delete-by time.
Walk me through pre-flight before I touch any AWS resource: (1) Verify AWS CLI is configured: `aws sts get-caller-identity`. The Account ID and Arn it returns — is this MY personal account, or a shared one? (Important: never create EKS in a shared account without permission.) (2) Install eksctl: `brew install eksctl` (macOS) or download the binary on Linux. Verify: `eksctl version` should print 0.x. (3) Pick the region. For training, prefer ap-southeast-1 (Singapore — closest latency for Vietnam) or us-east-1 (cheapest data transfer). Set: `export AWS_REGION=ap-southeast-1`. From here on, every command implicitly uses this region. (4) Pre-cost check: `aws ec2 describe-availability-zones --region $AWS_REGION` works → CLI + region OK. (5) Commitment: the cluster will live for at most 8 hours. Set a phone alarm RIGHT NOW: "Delete EKS cluster" at 17:00. Why a hard alarm? Because lab-day fatigue is real and "I'll do it after dinner" is how engineers get $200 surprises. (6) Final: list any existing EKS clusters in your account: `eksctl get cluster --region $AWS_REGION`. If anything is there from previous sessions, delete it before creating a new one.
The single most important AWS habit: every command names the region explicitly or relies on AWS_REGION env var. A command that succeeds in the wrong region creates orphaned resources you can't see and won't delete.
WARN: if aws sts get-caller-identity shows a company account or a shared account, STOP. Get explicit permission or switch to a personal training account before proceeding.
2
Build your project's Docker image and push it to ECR. The M1/M2 Mac trap is real — your image must be linux/amd64 or it will CrashLoopBackOff on the EKS node.
Walk me through ECR + Docker push, including the architecture trap: (1) Create the ECR repo (one per service): `aws ecr create-repository --repository-name [project-service-name] --region $AWS_REGION`. Output: a repositoryUri like 123456789.dkr.ecr.ap-southeast-1.amazonaws.com/myapp. (2) Login Docker to ECR: `aws ecr get-login-password --region $AWS_REGION | docker login --username AWS --password-stdin [account-id].dkr.ecr.$AWS_REGION.amazonaws.com`. Token expires in 12 hours — re-run if push starts failing. (3) Build for linux/amd64. If you're on M1/M2/M3 Mac, default `docker build` produces arm64 — EKS nodes are amd64 — Pod will CrashLoopBackOff with `exec format error`. The fix: docker buildx build --platform=linux/amd64 -t myapp:v1 . For Linux/x86 hosts: regular `docker build` is fine. (4) Tag for ECR: `docker tag myapp:v1 [account-id].dkr.ecr.$AWS_REGION.amazonaws.com/[repo-name]:v1`. (5) Push: `docker push [account-id].dkr.ecr.$AWS_REGION.amazonaws.com/[repo-name]:v1`. Verify in console or via `aws ecr list-images --repository-name [repo-name]`. (6) Repeat for every service in your project that you build yourself (not for Helm-installed 3rd-party services like Redis — those pull from public registries).
Verify image architecture before push: docker inspect myapp:v1 | grep Architecture. Should print "amd64". If it prints "arm64", you'll burn 30 minutes debugging a CrashLoopBackOff that's actually just wrong-arch.
Don't commit the ECR account ID to your repo — use an env var or pull from aws sts get-caller-identity. Account IDs aren't secret, but they're noise in PRs and make code less portable.
3
Create the EKS cluster the cost-safe way. eksctl create cluster with default flags will provision a NAT Gateway you don't need. Use a config file with the right knobs.
Help me write a cost-safe cluster.yaml for eksctl, then create the cluster: (1) The defaults to override: - 2 AZs is enough (eksctl default is 3 — saves data transfer) - 1 node, t3.medium, SPOT — not on-demand, not 3 nodes - Public subnets only, NO private subnets, NO NAT Gateway - Cluster + node group in the same managed node group config - Latest stable K8s version (1.30 or whatever current stable is) (2) Write cluster.yaml: apiVersion: eksctl.io/v1alpha5 kind: ClusterConfig metadata: name: project-[your-name]-lab region: ap-southeast-1 version: "1.30" availabilityZones: [ap-southeast-1a, ap-southeast-1b] vpc: nat: gateway: Disable # <-- saves $30/mo managedNodeGroups: - name: ng-spot instanceTypes: [t3.medium] spot: true desiredCapacity: 1 minSize: 1 maxSize: 1 privateNetworking: false # <-- nodes in public subnet, no NAT needed iam: attachPolicyARNs: - arn:aws:iam::aws:policy/AmazonEKSWorkerNodePolicy - arn:aws:iam::aws:policy/AmazonEKS_CNI_Policy - arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryReadOnly addons: - name: aws-ebs-csi-driver (3) Create: `eksctl create cluster -f cluster.yaml`. Wait time: 15–20 minutes — eksctl provisions the VPC, the EKS control plane, and the node group in sequence and prints progress. (4) While waiting, open the EKS console in another tab to watch the cluster status flip from CREATING to ACTIVE. Don't panic at long waits — VPC + node group provisioning is naturally slow. (5) Failure recovery: if `eksctl create` exits with an error, run `eksctl delete cluster --name [name] --region $AWS_REGION --wait` first to clean up any half-provisioned resources, then fix the config and retry. Never re-run `eksctl create` against a cluster that partially exists.
Two flags carry most of the cost savings: vpc.nat.gateway: Disable (saves $30/mo) and managedNodeGroups[].spot: true (saves 70% on EC2). With both set, an 8-hour lab is ~$1.10. Without either, the same lab plus 30 days idle is $108.
WARN: if eksctl create exits with an error, do NOT just retry — run eksctl delete cluster --name [name] --wait first to clean up partial state. Otherwise you accumulate orphaned resources that still cost money.
4
Connect kubectl to your new cluster. Verify the node is Ready and the EBS CSI driver loaded.
Walk me through kubectl auth and post-create verification: (1) Update kubeconfig: `aws eks update-kubeconfig --region $AWS_REGION --name project-[your-name]-lab`. This adds a new context to ~/.kube/config and switches to it. Verify: `kubectl config current-context`. (2) Verify nodes: `kubectl get nodes`. Expect: 1 node in Ready state, with INTERNAL-IP and EXTERNAL-IP both populated (because nodes are in public subnets). Roughly 90 seconds after cluster create completes. (3) Verify CNI + CoreDNS: `kubectl get pods -n kube-system`. Expect: aws-node, coredns, ebs-csi-* all Running. If something is Pending → describe it for the actual reason. (4) Common kubectl errors and what they mean: - "couldn't get current server API group list: ... no such host" → wrong region in update-kubeconfig - "An error occurred (ExpiredToken)" → AWS credentials timed out, re-auth - "error: You must be logged in to the server (Unauthorized)" → wrong AWS profile, the cluster was created by a different identity (5) The auth chain: kubectl reads ~/.kube/config → finds the EKS context → calls `aws eks get-token` → AWS returns a bearer token → API server validates it via IAM. If any link is wrong, you get Unauthorized.
If kubectl get nodes shows zero nodes 5 minutes after cluster create completes — the managed node group is still attaching. Wait 2 more minutes, then describe the node group in the console for status. Most often: ASG instance is launching.
5
Apply your Day 14 K8s manifests to EKS. Three things change vs minikube — image URLs, Service exposure, and how Secrets are sourced.
Help me adapt my Day 14 manifests for EKS: (1) Image URL: change `image: myapp:v1` (which worked on minikube via local docker daemon) to `image: [account-id].dkr.ecr.$AWS_REGION.amazonaws.com/myapp:v1`. The node IAM role from Q3 has `AmazonEC2ContainerRegistryReadOnly` — no `imagePullSecrets` needed. (2) Service exposure for the public-facing service: change `type: ClusterIP` to `type: LoadBalancer`. EKS auto-provisions a Classic ELB (5–10 min). Run: `kubectl get svc [name] -w` until EXTERNAL-IP populates with a hostname like a1b2c3.elb.ap-southeast-1.amazonaws.com. (3) Secrets — same `kubectl create secret generic` workflow as Day 14. For training that's fine. (In production: External Secrets Operator + AWS Secrets Manager, not today.) (4) ConfigMaps — apply unchanged. Database hostnames stay as Service DNS names like `postgres-svc.default.svc.cluster.local` — the Helm release in Q6 will publish that name. (5) Apply order: ConfigMap → Secret → Deployment → Service. Same as Day 14. Verify each: `kubectl get cm,secret,deploy,svc,pods`. (6) Common error: ImagePullBackOff with "401 Unauthorized" → node IAM role missing the ECR policy. Fix: add `AmazonEC2ContainerRegistryReadOnly` to the node group IAM role in IAM console, then `kubectl rollout restart deployment [name]`.
The ELB hostname takes 3–8 minutes to become reachable even after EXTERNAL-IP appears in kubectl get svc — the LB targets need to register and pass health checks. Don't curl immediately; wait until aws elb describe-instance-health shows InService.
6
Install your Day 15 Helm chart (Redis / RabbitMQ / PostgreSQL) on EKS. The values file needs one change vs minikube: storage.
Help me re-install my Helm chart on EKS: (1) On minikube you may have set `persistence.enabled=false` (Day 15) because minikube's default StorageClass was buggy. On EKS, the EBS CSI driver from Q3 provisions PVCs cleanly — so persistence can be enabled. (2) Update my-redis-values.yaml (or whichever chart): set `persistence.enabled=true`, `persistence.size=8Gi`, `persistence.storageClass=gp2` (default EKS StorageClass). (3) Install: `helm install my-redis bitnami/redis -f my-redis-values.yaml`. Verify: `kubectl get pvc` shows the PVC Bound (not Pending — Pending means the EBS CSI driver isn't running, see Q4). (4) Connect your application: same workflow as Day 15. Update your app's ConfigMap REDIS_HOST to the Helm-published service name, `kubectl rollout restart deployment [app]`. (5) End-to-end check: from inside your app pod, `kubectl exec -it [app-pod] -- redis-cli -h my-redis-master ping` → PONG. (6) Helm release survives node restarts because the release Secret is in the cluster. But it does NOT survive cluster delete — when you delete the cluster in Q8, the chart goes too. That's why we install fresh today.
EKS PVCs use EBS volumes — they're zonal. If your node is in ap-southeast-1a but the PVC was created in 1b (cross-AZ scheduling glitch), the pod can't mount it. Fix: kubectl delete pvc + reinstall, or use a single-AZ node group.
7
Curl your project from outside AWS — through the ELB, into the cluster, hitting your app, talking to the Helm-installed dependency. Real production-shaped traffic.
End-to-end verification — I want to prove the whole chain works: (1) Get the ELB hostname: `kubectl get svc [public-svc] -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'`. Returns: a1b2c3.elb.ap-southeast-1.amazonaws.com. (2) Health check first: `curl http://[elb-hostname]/health` from your laptop. Should return 200 within ~5 minutes of LB creation. If you get connection refused → wait. If 502 → backend pod not ready, check `kubectl get endpoints [svc]`. (3) Real application call — exercise actual logic that touches the Helm-installed dep: e.g. `curl -X POST http://[elb-hostname]/api/items -d '{"name":"test"}'` then `curl http://[elb-hostname]/api/items` — does the value persist? If yes, the full chain works: laptop → ELB → Service → Pod → app code → Redis/Postgres. (4) `kubectl logs [app-pod] --tail=20` while curling — confirm the request shows up in app logs. (5) `kubectl get all` — what does a fully healthy EKS deployment look like? Compare to your Day 14 minikube `get all` output: same shape, EKS adds the LoadBalancer Service with a real EXTERNAL-IP. (6) Take a screenshot of the curl output for your daily report.
If the ELB hostname returns 502 for one path but 200 for another → the public Service has no Endpoints for that backend. kubectl get endpoints must show the pod IP. If <none>, the Service selector doesn't match the Pod labels — same debug as Day 14 minikube.
8
Delete the cluster cleanly. Verify nothing was left orphaned. Write memory/eks-practice.md — your agent's playbook for the next EKS cluster.
The most important step. Don't skip the verification. Task 1 — Delete cleanly: (1) Uninstall Helm releases first (releases on a deleted cluster don't cost anything but it's good hygiene): `helm uninstall my-redis`. The PVC and EBS volume go with it. (2) `eksctl delete cluster --name project-[your-name]-lab --region $AWS_REGION --wait`. The `--wait` matters — it blocks until eksctl confirms every resource is gone. Without --wait you may exit before deletion completes. (3) Verify the cluster is gone: `eksctl get cluster --region $AWS_REGION`. Output should NOT list your cluster name. `aws eks list-clusters --region $AWS_REGION` should return an empty `clusters: []`. (4) Sweep for orphans (the part eksctl can miss if something went wrong): - EC2 → Volumes: any volumes tagged with your cluster name? If yes, the PVC didn't release cleanly — delete them manually. - EC2 → Load Balancers: any LBs from your Service of type LoadBalancer? Delete them manually if eksctl missed them. - VPC console: any VPC tagged `alpha.eksctl.io/cluster-name=[your-cluster]`? If yes, manually detach + delete. (5) Optional cleanup: `aws ecr delete-repository --repository-name [your-repo] --force` if you don't need the image again. ECR storage is cheap but the image you pushed today is dev-only. Task 2 — Write memory/eks-practice.md: 1. Create command (full): `eksctl create cluster -f cluster.yaml` + the file contents 2. Connect command: `aws eks update-kubeconfig --region X --name Y` 3. Image URL pattern: `[account].dkr.ecr.[region].amazonaws.com/[repo]:[tag]` 4. The cost stack: control plane + node + ELB hourly rate 5. Skip-list (what NOT to enable on a training cluster) — copy from today 6. Common errors + fixes — your real ones today, not generic 7. Delete sequence: helm uninstall → eksctl delete --wait → verify with `eksctl get cluster` (empty) + EC2 Volumes (no orphans) + EC2 Load Balancers (no orphans) + VPC tags (no eksctl VPC) 8. The "minikube vs EKS" diff table — image URL, Service type, persistence, secret source Format: tables, command blocks, no prose. This file is loaded into agent context, not read by a human.
HARD STOP: if eksctl delete cluster --wait exits with an error, DO NOT walk away. Re-run it once. If it still fails, open the EKS, EC2, and VPC consoles, find every resource tagged with your cluster name, and delete them manually. An orphaned NAT Gateway (if you didn't disable it) costs $30/month forever.
