Software App Fundamentals
When I receive an application to deploy, what is actually running — and why does the architecture shape everything I have to build around it?
The tools you choose as a DevOps engineer — pipelines, orchestrators, monitoring strategy — are all downstream of the application's architecture. A monolith and a microservice system require fundamentally different operations stacks. You cannot design infrastructure for a system you don't understand. Every production incident you ever face will require you to trace a request through components you did not write.

Discovery Chain — 7 questions, in order
How this works
Below are 7 questions. Each one has a purple prompt — that's the exact text you copy and paste into your Claude session.
The workflow for each question:

2.Copy the purple prompt below it

Important: The questions build on each other — Q3 only makes sense after you understand Q2. Don't skip ahead. Q7 directly generates your artifact file — save that output into memory/software-architecture.md.
1
You receive a URL returning 500 errors in production. You have no idea what the app does. What is the first thing you ask — and what tools trace the failing request from browser to database?
Context: I'm a DevOps engineer on-call. A URL is returning 500 errors. I have no prior knowledge of this app. Walk me through the exact diagnostic steps — what questions I ask the dev team and what CLI tools I run — to trace this request from browser to database.
2
The app has a frontend, a backend API (Node.js), and a database. What does a single "add to cart" action actually do — how many network calls, what data moves, and what could fail at each step?
Trace a single "add to cart" HTTP request through this stack: browser → Node.js API → database. Show me every network call, every piece of data that moves, and the specific failure mode at each step. Format as a numbered sequence.
3
The backend now calls three separate services (inventory, pricing, payment) before responding. What failure modes exist that didn't exist in the monolith — and what happens to the cart if the pricing service is slow?
I've split a monolith into three microservices: inventory-service, pricing-service, payment-service. Each is a separate deployment. Compare the failure modes of this architecture versus the monolith. Be specific: what happens to the cart request if pricing-service responds in 8 seconds instead of 200ms?
Key concept you're discovering: blast radius, cascading failure, timeout propagation
4
The team uses a message queue between the order service and the fulfillment service. Why not a direct API call? What breaks under load if they had used one?
Explain why an e-commerce system uses a message queue (like SQS or RabbitMQ) between order-service and fulfillment-service instead of a direct HTTP call. Show me concretely what breaks under 1000 orders/minute if they had used a direct call — and how the queue solves each problem.
Key concept: decoupling, backpressure, at-least-once delivery
5
Redis sits between the API and the database. Every cache miss hits the database. The database is at 95% CPU. How do you decide: scale the database or fix the cache?
The database is at 95% CPU. Redis is in front of it. I need to decide: scale the database (vertical or horizontal) or fix the cache (TTL, eviction policy, cache warming). What are the three diagnostic questions I answer first — and what does each answer tell me about which action to take?
Key concept: caching strategy, cache invalidation, the trade-off between cost and correctness
6
The team wants to add a new microservice. They ask you to set up CI/CD for it. Compared to the monolith pipeline you already have — what new concerns appear?
I already have a CI/CD pipeline for a monolith. Now I need to add a pipeline for a new microservice that's part of a distributed system. List the specific new concerns that didn't exist for the monolith: deployment order, service discovery, distributed tracing, and independent rollback. One paragraph each.
7
You sit down to write your memory file. What are the five architectural patterns a DevOps engineer must recognize on sight — and what does each imply about the operations work required?
I'm writing memory/software-architecture.md for my DevOps agent. List the five application architecture patterns I must recognize instantly (monolith, microservices, event-driven, serverless, and one more of your choice). For each: one sentence description + exactly what operations work it implies — pipeline complexity, scaling model, observability requirements, and rollback strategy.
This prompt builds your artifact directly — save the output into your memory file.
