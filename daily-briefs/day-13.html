Kubernetes Intermediate: Ingress, ConfigMap, Secret, HPA
How do you manage external access, configuration, sensitive data, and automatic scaling — the four operational concerns that appear in every production cluster?
Yesterday's Deployment and Service got your app running. That is a toy cluster. Production adds four concerns that a bare Deployment + Service cannot solve: external traffic needs a single entry point with hostname and path routing (Ingress), configuration must change per environment without rebuilding the image (ConfigMap), credentials must not sit in plain text in manifests (Secret), and replica count must react to load automatically (HPA). These four objects are the gap between a cluster that runs an app and a cluster that serves production.

Concept — the four production objects
Each object fills a specific gap that a bare Deployment + Service leaves open. Read this before touching a manifest — the question is not "what does this object do," it is "what problem does it solve that I would otherwise solve badly by hand."
🛡
1. Ingress
Single entry for many services
NodePort gives every service a different high port. With 20 services you have 20 firewall rules, 20 DNS entries, and no hostname-based routing. An Ingress is a set of rules ("host api.example.com goes to api-svc, path /admin goes to admin-svc") enforced by an Ingress controller — a real Pod running nginx or traefik. The Ingress object is the config; the controller is the process that reads it.
📄
2. ConfigMap
Non-sensitive config, decoupled
Feature flags, database hostnames, log levels, base URLs — values that differ between dev/staging/prod. Hardcoding them in the Deployment means a different manifest per environment. A ConfigMap lives separately and is mounted into the container two ways: envFrom (every key becomes an env var) or volumeMount (every key becomes a file). Same image, different config, nothing rebuilt.
🔒
3. Secret
Sensitive values, access-controlled
Shape-wise identical to a ConfigMap. The difference is governance, not cryptography. Secret values are stored base64-encoded in etcd — that is encoding, not encryption. What makes a Secret "secret" is RBAC: only principals with get secret permission can read it. For real encryption at rest, enable encryption providers on the API server; for real production secrets, use a dedicated secret manager (Vault, AWS Secrets Manager) with the Secrets Store CSI driver.
📈
4. HPA
Replica count reacts to load
A Horizontal Pod Autoscaler watches CPU (or custom metrics) and adjusts a Deployment's replica count between minReplicas and maxReplicas. It requires metrics-server — without it, HPA reports unknown for utilization and does nothing. HPA scales Pods, not nodes. When Pods go Pending because no node has room, that is the cluster autoscaler's job — a separate system that adds nodes.
ConfigMap vs Secret — the single rule that matters
ConfigMap
Feature flags, hostnames, log levels, base URLs, file paths. Safe to appear in logs, safe in a PR diff.
Secret
Database passwords, API keys, TLS private keys, OAuth tokens. Would cause an incident if leaked to a log, PR, or screenshot.
Rule: if leaking the value to CI logs would require an incident review, it goes in a Secret. If it would not, a ConfigMap is enough. Do not mix the two in the same object.
The production gap. A Deployment + Service runs an app. Adding Ingress + ConfigMap + Secret + HPA is what makes the cluster operable: one hostname for users, one config knob per environment, credentials handled under access control, and capacity that reacts to real load. Every production K8s deployment you will encounter uses these four.
Discovery chain — 8 questions
1
You have three microservices behind three NodePort Services. A load balancer forwards traffic to three high ports. A fourth service means another port, rule, and DNS entry. What is the operational problem — and what does an Ingress controller give you instead?
I have 3 microservices exposed as NodePort Services on ports 30081, 30082, 30083. I am about to add a 4th and a 5th. Explain the scaling problem with NodePort-per-service: (1) what breaks operationally at 10+ services — ports, firewall rules, DNS, TLS certs, (2) what an Ingress controller is — make it explicit that it is a REAL Pod running nginx or traefik, not an abstract concept, (3) what an Ingress resource is vs the Ingress controller — the config rule vs the process that enforces it, (4) the two forms of routing an Ingress supports: host-based (api.example.com vs admin.example.com) and path-based (/api vs /admin), with a concrete example of each. Keep it under 300 words.
The Ingress resource is just YAML. The Ingress controller is a running Pod (you installed it via minikube addons enable ingress). Confusing these two is the single most common Day-13 mistake.
2
Write an Ingress with path-based routing: /api → api-service, / → frontend-service. After applying, frontend works but /api returns 404. What three things do you check — in order?
I applied this Ingress with two path rules (/api to api-svc:8080, / to frontend-svc:3000). Frontend loads correctly but /api returns 404 from the nginx Ingress controller. Give me a 3-step debugging checklist in the right order — cheapest to most expensive check first: (1) what to look at in the Ingress resource itself (pathType, backend service name, port name vs port number), (2) what to verify on the Service (does the Service exist, does it have Endpoints — run kubectl get endpoints [svc-name]), (3) what to verify on the Pods (labels match the Service selector, container listening on the right port). For each step, give me the kubectl command and what a good vs bad result looks like.
pathType: Prefix is the one most people get wrong — Exact means "/api" only, not "/api/users". And check kubectl get endpoints api-svc — if it shows <none>, the Service selector does not match any Pods and no amount of Ingress debugging will help.
When your Ingress debugging goes past 15 minutes, run kubectl logs -n ingress-nginx -l app.kubernetes.io/name=ingress-nginx --tail=50 — the controller logs show the actual request routing decision.
3
Your Deployment has DATABASE_HOST and LOG_LEVEL hardcoded. Values differ in dev, staging, prod. What is the problem with this at scale, and how does a ConfigMap decouple configuration from the Deployment?
Show me the transformation from "config hardcoded in Deployment" to "config in a ConfigMap": (1) First, write a minimal Deployment YAML with env vars DATABASE_HOST=db.internal and LOG_LEVEL=info hardcoded inline (2) Explain the scaling problem when you need 3 environments (dev, staging, prod) with different values — how many manifests do you end up maintaining and what goes wrong at PR-review time (3) Now write a ConfigMap YAML holding DATABASE_HOST and LOG_LEVEL (4) Rewrite the Deployment to pull env vars from the ConfigMap TWO ways: (a) envFrom.configMapRef (whole ConfigMap becomes env vars), (b) env.valueFrom.configMapKeyRef (one specific key). Show both. (5) One sentence on when to pick envFrom vs valueFrom Keep each YAML under 25 lines. I want to read them side by side.
envFrom is the one-line shortcut when you want the whole ConfigMap injected. env.valueFrom.configMapKeyRef is when you only need one specific key or need to rename the variable. Both are valid; they are not redundant.
4
You move non-sensitive config to a ConfigMap. The database password is still in plain text in the Deployment. You create a Secret. What IS a Secret — and why is base64 NOT encryption?
Be explicit and blunt on this one — this is the single most misunderstood K8s concept. Answer: (1) What a Secret is — how it differs from a ConfigMap (shape is similar, governance is different) (2) How values are stored — base64-encoded in etcd. Show me with: echo -n "supersecret" | base64 and the reverse with base64 -d. Make it concrete that anyone with kubectl get secret and base64 -d sees the value. (3) Why base64 is encoding, not encryption — one sentence on what each word means and why people confuse them (4) What actually protects a Secret in practice: RBAC (who can kubectl get secret), etcd encryption-at-rest (cluster setting, off by default), and external secret stores for production (Vault, AWS Secrets Manager, Azure Key Vault via CSI driver) (5) The two mount methods — env var vs file (volumeMount) — and one tradeoff each. Hint: env vars appear in process listings and crash dumps; files do not. End with: if someone says "Secrets are encrypted in Kubernetes," what do you correct them on?
Mounting a Secret as a file (volumeMount) is generally safer than as an env var — env vars show up in /proc, crash dumps, and accidental env prints. Files stay on disk inside the container only.
5
A security audit flags that any developer with kubectl get secret can read database passwords. What Kubernetes mechanism restricts this — and how is it different from using AWS Secrets Manager or Vault?
Explain the two layers of secret access control in Kubernetes and why production teams usually run both: (1) RBAC inside the cluster — what Role, ClusterRole, RoleBinding, ClusterRoleBinding are, and how a Role that excludes "secrets" from its rules prevents a developer from reading them. Show me a minimal Role YAML that grants get/list on pods and deployments but NOT secrets, in a namespace. (2) External secret management — AWS Secrets Manager, HashiCorp Vault, Azure Key Vault. What the Secrets Store CSI Driver is and how it pulls a secret from an external store and mounts it into the Pod at runtime — so the value never lives in etcd at all. (3) When each approach is enough: RBAC alone is enough for a small team with homogeneous trust. An external store is needed when: auditors require rotation, multiple clusters share credentials, or the secret is managed by a separate team (DBA, security). Keep this practical — I need to know which approach to reach for when.
The CSI driver is the modern default for cloud deployments. It means Secrets become a cache of an external source of truth, not the source itself. Rotating a secret in AWS Secrets Manager causes every Pod using it to pick up the new value on restart — no kubectl needed.
6
Your app is pegging 90% CPU. You could run kubectl scale deployment --replicas=10, but you are going on holiday. What does an HPA do, what does it need, and what are its three main knobs?
Walk me through configuring a Horizontal Pod Autoscaler end-to-end for a Deployment named "web": (1) Prerequisite: the metrics-server must be running. Command to verify: kubectl top pods. Explain what happens if metrics-server is NOT installed (HPA reports utilization as unknown and does nothing — the most common HPA failure mode) (2) Resource requests matter — HPA targets a percentage of spec.containers.resources.requests.cpu. If the container has no CPU request, HPA cannot compute utilization. Show me a Deployment snippet with a cpu request of 100m. (3) Write the HPA YAML (autoscaling/v2) for Deployment web: minReplicas: 2, maxReplicas: 10, target CPU 70% averageUtilization. (4) Explain the three parameters in one sentence each: minReplicas, maxReplicas, targetAverageUtilization. (5) What is the scale-down cooldown and why does it exist? (Pods are killed slower than they are added — prevents flapping between sizes under bursty traffic.) (6) The command to watch scaling happen live: kubectl get hpa -w
Metrics-server is disabled by default on most clusters. On minikube: minikube addons enable metrics-server. Then wait ~60 seconds and verify with kubectl top pods — if you see values, HPA can work. If you see an error, HPA will report unknown forever.
Actually trigger a scale event today. Run a load generator, watch kubectl get hpa -w, see replica count climb, stop the load, see it shrink. That observation is the entire point of this question.
7
HPA scales to 10 replicas. 3 Pods are Pending — no node has enough CPU. HPA has hit the limit of what it can do. What is cluster-level autoscaling, and how is it different from HPA?
Explain the second layer of autoscaling that HPA alone cannot provide: (1) The problem: HPA changed replica count from 2 to 10, but the cluster has 2 nodes with 4 CPU each — not enough to schedule all 10 Pods. 3 Pods show Pending. kubectl describe on a Pending Pod shows "0/2 nodes available: insufficient cpu." (2) What the Cluster Autoscaler does — watches for Pending Pods and adds nodes to the cluster (by talking to the cloud provider: EC2 Auto Scaling Group, Azure VMSS, GCP instance group). Explain the feedback loop: new node joins -> kubelet reports ready -> scheduler places the Pending Pod. (3) How HPA and Cluster Autoscaler compose: HPA scales the Deployment (more Pods needed) -> scheduler tries to place them -> if no room, Pods go Pending -> Cluster Autoscaler adds a node -> Pods get placed. It is two independent loops. (4) Managed-cluster names for this: EKS = Managed Node Groups + Cluster Autoscaler (or Karpenter), AKS = Cluster Autoscaler, GKE = Node Auto-Provisioning. (5) What minikube does for this question — nothing, because it is single-node. Note that this is a concept you will see in real clusters in Week 5.
One-sentence summary to memorize: HPA scales Pods. Cluster Autoscaler scales Nodes. They are different controllers solving different problems, and production clusters run both.
