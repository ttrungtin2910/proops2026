Concept — the five problems orchestration solves
Docker Compose is enough for one machine. The moment you need more than one server — for traffic, for reliability, or for zero-downtime deploys — five specific problems appear. Every orchestrator (Swarm, Kubernetes, Nomad) exists to solve these same five problems. Read this before touching any commands.
📈
1. Scheduling
Which machine runs which container?
With 3 servers and 9 containers, you must decide manually where each container runs — and track available CPU and memory across all servers. One wrong placement and a server goes OOM. An orchestrator tracks resource availability across every node and places containers automatically.
❤
2. Health monitoring
What happens when a container crashes?
A container crashes at 3am on server 2. No alert fires. It stays down until someone checks in the morning — 6 hours of partial outage. An orchestrator runs health checks continuously and restarts failed containers automatically, on the same node or a different one.
🔌
3. Service discovery
How do containers find each other?
Container IPs change on every restart and every reschedule. Hardcoding IPs breaks immediately. An orchestrator gives each service a stable DNS name — your API always reaches the database via db, regardless of which node it is running on or what IP it has today.
🔁
4. Rolling deployments
How do you deploy without downtime?
Updating 10 servers manually means SSH-ing into each one, stopping the old container, starting the new one — while some servers serve old traffic and others serve new. An orchestrator replaces containers in controlled batches, keeps old ones running until new ones pass health checks, and stops automatically if something is wrong.
⇅
5. Scaling
How do you handle more traffic?
Adding a second copy of your API on another server means updating load balancer config, tracking which instances exist, and cleaning up when traffic drops. An orchestrator does this with one command — or automatically based on CPU usage — and removes instances cleanly when they are no longer needed.
Today you will experience problem #2 directly. You will run 2 replicas of a service in Docker Swarm, kill one container manually, and watch the orchestrator restart it — automatically, in under 10 seconds. That one demo makes all five problems real. Keep this list in mind as you work through Q6–Q7.

Part 1 — Deploy to EC2
1
Provision a fresh t3.micro EC2 in the default VPC. SSH in, install Docker and Docker Compose. What is the minimum setup needed to get a clean server ready to run containers?
I need to provision a fresh EC2 instance and get it ready to run Docker containers. Walk me through: (1) the AWS Console steps to launch a t3.micro using Amazon Linux 2023 (or Ubuntu 22.04) in the default VPC and default subnet — no new VPC needed, (2) what Security Group rules to add at launch — SSH (port 22) from my IP only, plus the ports my services use, (3) once I SSH in, the exact commands to install Docker, add my user to the docker group, start and enable the Docker daemon, and install Docker Compose v2 as a plugin, (4) the two commands to verify Docker and Compose are working before I do anything else. Keep it minimal — I do not need anything beyond what is required to run docker compose up.
Use Amazon Linux 2023 — it has the cleanest Docker install path with dnf install docker. After adding yourself to the docker group, run newgrp docker to apply without logging out. Open only the ports your services actually need in the Security Group — not a wide-open 0.0.0.0/0 range.
No Elastic IP, no NAT Gateway, no new VPC. Default VPC + default subnet + Security Group is everything you need for today.
2
Your images are on Docker Hub (you pushed them on Day 9). Pull them onto the EC2. Before you can run the stack, you also need your .env file — but it is not in the repo. How do you get it onto the server securely?
I need to get two things onto my EC2 to run my stack: (1) my Docker images from Docker Hub, and (2) my .env file which contains database passwords and API keys — it is gitignored so it is not in the repo. Explain: (a) the docker pull command to pull a specific image and tag from Docker Hub, (b) two safe ways to get my .env file onto the EC2 without putting secrets in the repo or typing them into the terminal history — what is scp and what is the heredoc approach, (c) what permissions should the .env file have on the server and why.
The simplest approach for a training environment: use scp to copy your local .env file directly to the EC2. Run: scp -i your-key.pem .env ec2-user@[public-ip]:~/ from your local machine.
3
You copy your docker-compose.yml to the EC2 and run docker compose up. Some containers start, some fail. What are the three most common reasons a Compose stack that works on your laptop fails on a fresh EC2?
I copied my docker-compose.yml to an EC2 and ran docker compose up. Some services started but others failed. Before I look at specific error messages, help me build a mental checklist: what are the 3 most common reasons a Compose stack that works perfectly on a developer laptop fails on a fresh Linux server? For each reason: (1) what the symptom looks like in docker compose logs, (2) the root cause, (3) the fix. Focus on: image availability, environment variables, and port/user permission issues.
The three usual suspects: (1) image not pulled or wrong tag — docker compose pull first, (2) .env file missing or in the wrong directory, (3) a port below 1024 that needs root on Linux. Read docker compose logs [service-name] — the error is almost always in the first 10 lines.
4
The stack is running. You try to reach your API from a browser using the EC2 public IP — connection refused. The container is up. What is the missing piece?
My docker compose up shows all services as running and healthy. I try to open http://[ec2-public-ip]:3000 in a browser — connection refused. The container is definitely running and listening on port 3000 inside the container. Walk me through: (1) what is an EC2 Security Group and why does it block my request even though the container is running, (2) what inbound rule do I need to add — what port, what protocol, what source CIDR — to allow HTTP access to my API, (3) after I add the rule, is there anything else that could still block the connection — what does docker compose ports mapping do and what does it mean if I see 127.0.0.1:3000 vs 0.0.0.0:3000 in docker ps output?
Two separate systems control access: (1) AWS Security Group — controls traffic at the network level before it reaches the instance, (2) Docker port binding — controls what interface the container listens on inside the instance. Both must allow the traffic. Add the Security Group inbound rule in the AWS Console, then check docker ps to confirm the port binding shows 0.0.0.0:3000.
Only open the ports your services actually use. Do not open port 0-65535 to 0.0.0.0/0 — that defeats the purpose of a security group.
5
Your services are reachable. But if you close the SSH terminal, the stack stops. How do you keep it running after you disconnect — and how do you verify it is still up after reconnecting?
My docker compose stack is running and accessible on the EC2 public IP. But when I close my SSH terminal the stack stops. I need it to keep running after I disconnect. Explain: (1) why closing the terminal stops the processes — what is a foreground process vs a background process in Linux, (2) what does docker compose up -d do and how is it different from running without -d, (3) after I disconnect and reconnect, what commands do I run to verify the stack is still running and healthy, (4) if the EC2 reboots, will the containers restart automatically — and if not, what is the simplest way to make them start on boot?
docker compose up -d is the answer to disconnecting safely. After reconnecting: docker compose ps to check status, docker compose logs --tail=20 to check for errors. For auto-restart on reboot: restart: unless-stopped in each service definition in your compose file.
Part 2 — Swarm Intro
6
Initialize Docker Swarm on your EC2 and create a service with 2 replicas. Then kill one container manually. Watch what happens.
I want to see Docker Swarm's automatic recovery in action on my EC2. Give me the exact commands to: (1) initialize a single-node Swarm with docker swarm init, (2) create a service called "web" using the nginx image with 2 replicas, (3) list the running tasks of the service to see both replicas, (4) get the container ID of one replica from docker ps, (5) kill that container with docker kill [container-id], (6) immediately run docker service ps web again to see what the Swarm manager does. Explain what I should see in the output after step 6 — specifically what the CURRENT STATE and DESIRED STATE columns will show.
You will see one task show Failed and a new one appear as Running almost immediately. The DESIRED STATE column always says Running — the manager's job is to make reality match that. This is the reconciliation loop that Kubernetes also uses.
This takes about 5 minutes to run. Do it — watching it happen is the entire point of Part 2. Screenshots of the before/after output are good evidence for your daily report.
7
You just saw automatic recovery. What are the four other problems orchestration solves beyond health monitoring — and where does Docker Compose fall short on each one?
I just watched Docker Swarm automatically restart a container I killed. That demonstrated one orchestration problem: health monitoring and recovery. Explain the four other problems that appear when you run containers across multiple servers — not just one EC2: (1) scheduling — who decides which server runs which container, (2) service discovery — how containers find each other when IPs change, (3) rolling deployments — how you update without downtime, (4) scaling — how you add more copies under load. For each one: what breaks if you try to do it manually with just Docker Compose, and what does an orchestrator do instead? Keep it concrete — one paragraph each.
