Agent Checkpoint: Memory Review
Are your memory files good enough to actually help you — or are they just storage?
You have been writing memory files since Day 1. Today is the first time you will test whether they work. A memory file is not notes you write for yourself — it is context you provide to Claude so it can give you specific, accurate answers. A shallow memory file produces generic answers. A deep one produces answers that reference your exact setup, your actual commands, your specific decisions. Today you learn how to tell the difference, and how to fix the files that are not good enough yet. This method — test, find the gap, improve, re-test — is the quality loop you will run every checkpoint from here.

Concept — shallow vs deep memory files
Every memory file you write feeds Claude's answers the next time you ask about that topic. The quality of the file determines the quality of the answer. Here is what the difference looks like:
Shallow memory file
Looks done. Doesn't work.
# Docker

## What it is
Docker is a containerization tool.
It packages apps and dependencies together.

## Commands
- docker build
- docker run
- docker ps
- docker stop

## Compose
Docker Compose runs multiple containers.
Ask Claude "how do I debug a container that crashes on startup?" with this file — you get a generic answer that could apply to anyone.
Deep memory file
Specific. Actionable. Yours.
# Docker

## Debug: container exits immediately
1. docker logs <name> — read the last error
2. docker run -it <image> sh — enter manually
3. Common cause: CMD tries to connect to DB
   before it is ready — add healthcheck

## My Compose stack (mock project)
Services: api-gateway, user-service, task-service
Each service connects to its own DB container
Service names must match app env vars exactly

## Mistakes I already made
- Used localhost instead of service name → fixed
Ask Claude the same question — it references your stack, your commands, your specific mistake. Useful.
Three questions that reveal whether a memory file is deep enough:
Is it specific to you? Does it reference your actual project, your actual commands, your actual decisions — or is it generic enough to describe any developer?
Does it include commands? Not just "use docker logs" — the actual command with flags you run, and what you look for in the output.
Does it capture what went wrong? Mistakes you hit, problems you solved, things that surprised you — this is where half the value lives, and most people skip it entirely.
Discovery chain — 7 questions, 2 files audited
Q1–Q5 focus on memory/docker.md. This is your most recent file — you wrote it yesterday, so the content is fresh and the gaps will be obvious. Q6 is where you choose a second file and repeat the same method. Q7 is the output.
1
Before testing anything, ask Claude to describe what a useful memory file looks like from its perspective — what information in a file actually helps it give better answers?
I am building a memory file system for a DevOps AI agent. The files live in a memory/ folder and I paste their contents at the start of relevant sessions. From your perspective as the AI that reads these files: what kind of information in a memory file actually helps you give better, more specific answers? What are the top 3 types of content that make the biggest difference — and what are 3 things people commonly write that look useful but actually don't help?
Read this answer carefully. Claude will describe specificity, commands with context, and documented decisions as high-value. "Topic overviews" and "tool descriptions" as low-value. This is the rubric you will apply to your own files in the next questions.
2
Open memory/docker.md and read it fully. Before running any test — what do you think its weakest section is, and why?
Here is my current memory/docker.md: [paste the full contents of memory/docker.md here] Based on the rubric we just discussed — specificity, commands with context, documented decisions — identify the single weakest section. Explain specifically what is missing: is it (a) depth — the concept is named but not explained, (b) specificity — no real commands or examples from my actual work, or (c) coverage — a whole topic from my learning is absent entirely?
You are diagnosing before fixing. Do not rewrite anything yet — just understand what is wrong and why. A correct diagnosis leads to a correct fix. A skipped diagnosis leads to a rewrite that still fails the test.
Replace [paste the full contents of memory/docker.md here] with the actual file contents before sending. Do not send the prompt with the placeholder still in it.
3
Now run the real test: ask Claude a Docker question that your memory file should be able to answer. Evaluate whether the answer is specific to you, or generic enough to apply to anyone.
Using only the information in memory/docker.md that I just shared, answer this question as if you are my personal DevOps assistant who only knows what is in that file: "I have a container that crashes immediately after docker compose up. Walk me through exactly how to debug it — what commands do I run, in what order, and what do I look for in the output of each one?" Do not use any general Docker knowledge outside of what is in the file. If the file does not have enough information to answer specifically, tell me exactly what is missing.
This is the test. If the answer sounds like a Docker tutorial — generic steps that could come from any blog post — the file failed. If the answer references your actual stack, your specific services, your actual commands, it passed. Be honest about which one you got.
4
The test revealed a gap. Now fix it — rewrite the weakest section so it passes the same test when you run it again.
Based on the gap you identified, help me rewrite that section of memory/docker.md. The rewritten section should: 1. Reference my specific project setup (services: [list your services from your DOP]) 2. Include the exact commands I would run, with the real flags, not placeholder syntax 3. Include at least one "mistake I made / thing that surprised me" entry — something I learned the hard way this week 4. Be short enough to actually read — no walls of text, use bullets and short lines Write the replacement section only, not the whole file. I will paste it in myself.
After Claude writes the replacement — read it before you paste it in. Does it actually reference your project? Are the commands real? If it still looks generic, push back: "this still sounds generic — add my specific service names and the actual error I hit."
5
Paste the improved section into memory/docker.md and re-run the same test from Q3. Did the answer get materially better?
Here is my updated memory/docker.md after the rewrite: [paste the updated file contents here] Run the same test as before: using only this file, answer "I have a container that crashes immediately after docker compose up — walk me through exactly how to debug it." Compare this answer to the one you gave before the rewrite. What specifically improved, and is there still a gap?
You are looking for a materially better answer — not just slightly different wording. If the improvement is small, the rewrite addressed the wrong thing. Ask Claude what the remaining gap is and fix it. If the answer is genuinely specific and actionable now, the file passed.
A file that passes the test does not mean a perfect file — it means good enough to be useful. "Good enough to be useful" is today's standard. Perfection is a future checkpoint's job.
6
Pick one other memory file you wrote — the one you think is weakest — and run the full audit on it: test it, find the gap, fix the weakest section, re-test.
I want to audit another memory file using the same method we used for docker.md. Here is my [file name — e.g. memory/git.md]: [paste the full contents] Step 1: Identify the weakest section — depth, specificity, or coverage gap? Step 2: Give me a test question this file should be able to answer — something real I would actually ask in a working session. Step 3: Answer the test question using only the file contents. Tell me if the answer passes or fails. Step 4: Write the replacement for the weakest section, same standards as before — specific to my setup, real commands, at least one "lesson learned" entry.
Which file should you pick? Whichever one you least want to look at — that feeling usually means it is the one that got written the fastest and has the least depth. Common candidates: networking-basics.md, cloud-concepts.md, or git.md.
7
Write memory/gap-analysis.md — a simple table recording what you found today. This becomes your agenda for the Day 20 checkpoint.
Help me write memory/gap-analysis.md. It should be a simple markdown table with these columns: | File | What is weak | Gap type | Plan to fix | Fix by | Fill in one row for each file I audited today (minimum 2 rows). Then add one row for any other memory file you think has a likely gap based on what you know about my Week 1-2 learning — even if I haven't audited it yet. Gap type should be one of: depth / specificity / coverage / missing entirely Plan to fix should be one sentence — specific. Fix by: Day 20 (the next checkpoint). Keep it short. This is a working document, not a report.
This file is not graded on completeness today. Its job is to give future-you (and your mentor) a clear picture of where the gaps are. Three honest rows are more valuable than ten vague ones.
