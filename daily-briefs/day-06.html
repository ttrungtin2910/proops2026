AWS Hands-on: EC2, S3, VPC, CLI
How do you provision, connect to, and operate the core AWS building blocks (EC2, S3, VPC) that form the foundation of almost every cloud architecture?
EC2, S3, and VPC are the three primitives everything else in AWS is built on. A DevOps engineer who cannot provision a server, configure its network, and connect to it from the CLI is blocked before they can deploy anything. The AWS CLI is equally critical — manual console clicks don't belong in automation pipelines, and every real DevOps workflow eventually scripts its cloud operations. Day 5 gave you the theory. Discovery Chain — 8 questions, one AWS stack
Today's 8 questions follow a single scenario — provisioning a server, connecting to it, configuring storage, locking down the network, and scripting the whole thing with the CLI. Copy each purple prompt, paste into Claude, read the answer carefully, then rephrase the key idea in your own words before moving on.
1
You are asked to provision a server in AWS for a new application. Before clicking anything, what questions must you answer first — region, instance type, OS, and network?
I'm a DevOps engineer and I need to provision an EC2 instance for a new application. Before I touch the console or CLI, walk me through: (1) the four decisions I must make first — region, instance type, AMI (OS), and VPC/subnet — and what factors drive each decision, (2) where in the AWS console I find each option and what the defaults are, (3) the AWS CLI equivalent: show me the exact `aws ec2 run-instances` command with all required parameters, explain each flag. I want to understand the decisions, not just click through a wizard.
Region = latency + compliance. Instance type = workload profile. AMI = OS. VPC = network boundary.
2
You launch the EC2 instance. It shows "running" but you cannot SSH into it. What are the four things that must be correctly configured for SSH to work?
I launched an EC2 instance and it shows "running" in the console, but I cannot SSH into it. Walk me through the four things that must all be correct for SSH to work: (1) key pair — how it's created, where the private key lives, what happens if you lose it, (2) security group — what inbound rule is needed for SSH, what port, what source CIDR, (3) public IP — when an instance gets one vs when it doesn't, what an Elastic IP is, (4) route to the internet — what the route table must contain, what an Internet Gateway does. For each one, show me the AWS CLI command to check the current state and the command to fix it. This is a real troubleshooting scenario I need to be able to resolve in 5 minutes.
SSH troubleshooting is the most common Day 1 AWS problem. Master this flow.
3
You SSH in successfully. The app needs to read a config file from S3. What IAM concept lets EC2 access S3 without hardcoding credentials — and why is hardcoding a critical mistake?
I'm SSHed into my EC2 instance and the application needs to read a configuration file from S3. Explain: (1) why I should NEVER put AWS access keys in the application code or environment variables on the instance — what specific security risk this creates, (2) what an IAM Role is and how an Instance Profile works — the mechanism by which EC2 gets temporary credentials automatically, (3) the exact steps to create an IAM role with S3 read access and attach it to a running EC2 instance — both via console and via CLI (`aws iam create-role`, `aws ec2 associate-iam-instance-profile`), (4) how the application on the instance uses these credentials — what the SDK does behind the scenes with the instance metadata service.
IAM Roles + Instance Profiles = the correct way to give EC2 access to other AWS services
4
You run aws s3 ls from EC2 and get "Access Denied." What does this tell you about the IAM role or policy, and what is the minimum permission needed?
I'm on my EC2 instance and I run `aws s3 ls` but I get "Access Denied." Walk me through: (1) what this error means — is it an authentication problem or an authorization problem, and how to tell the difference, (2) how to check what IAM role is attached to this instance (`curl http://169.254.169.254/latest/meta-data/iam/security-credentials/`), (3) how to read the IAM policy attached to the role and identify exactly what's missing, (4) what the minimum S3 permission looks like for listing a specific bucket and reading specific objects — write the exact JSON policy document, (5) why "s3:*" is wrong and what "principle of least privilege" means in one sentence. Show me the CLI commands for each diagnostic step.
Least privilege: grant exactly what's needed, nothing more. Debug by reading the policy.
5
You upload a config file to S3 with aws s3 cp. The upload succeeds but the app can't find the file. What are two possible explanations?
I ran `aws s3 cp config.json s3://mybucket/` and it succeeded. But my application says it can't find the file at that path. Explain: (1) two possible causes — wrong S3 key path vs wrong bucket, and how S3 "paths" actually work (S3 has no folders — it's flat key-value storage with prefix-based display), (2) how to verify the file is actually where the app expects it using `aws s3 ls s3://mybucket/ --recursive`, (3) the difference between `aws s3 cp` and `aws s3api put-object` — when you need the lower-level API, (4) how to download the file back to verify its contents: `aws s3 cp s3://mybucket/config.json -`. Show me the exact commands for each step.
S3 has no folders — just keys. "directory/file.json" is a single key string, not a path.
6
The app is working. Now lock down the network: port 443 from the internet, port 5432 to a database in the same VPC. What VPC components do you configure?
My application is running on EC2 and I need to lock down the network. Requirements: (1) accept traffic on port 443 (HTTPS) from the internet, (2) allow outbound to a PostgreSQL database on port 5432 in a private subnet in the same VPC. Explain: (1) what a security group is — stateful firewall, inbound vs outbound rules, how "stateful" means return traffic is automatic, (2) the difference between a public subnet and a private subnet — route table, Internet Gateway, NAT Gateway, (3) the exact security group rules I need — write the JSON or CLI commands for both the web server SG and the database SG, (4) why the database should be in a private subnet and what happens if it's accidentally in a public subnet. Give me the full VPC design with two subnets.
Public subnet = route to Internet Gateway. Private subnet = route to NAT Gateway (or no internet at all).
7
A teammate needs the same setup in a new environment. What AWS CLI commands reproduce the EC2 + S3 + security group setup — and why is this the first step toward infrastructure as code?
A teammate needs to replicate my setup (EC2 instance + S3 bucket + security group + IAM role) in a new AWS account. I don't want to walk them through the console. Write me: (1) the complete sequence of AWS CLI commands to reproduce the entire setup from scratch — VPC, subnet, internet gateway, route table, security group, key pair, EC2 instance, S3 bucket, IAM role, instance profile — in the correct order, (2) explain why the order matters (you can't launch EC2 before the VPC exists), (3) why this script is the first step toward infrastructure as code — and what Terraform and CloudFormation do that a bash script of CLI commands does not. Give me the real commands, not pseudocode.
CLI scripts = reproducible but fragile. IaC tools add state tracking, dependency graphs, and drift detection.
8
Write memory/aws.md — the five AWS concepts every DevOps engineer must know before they can do anything else in AWS.
I'm writing memory/aws.md for my DevOps agent. Generate the full file with these 5 sections: 1. EC2 instance lifecycle — instance states (pending, running, stopping, stopped, terminated), how to launch/stop/terminate via CLI, what persists across a stop (EBS volume, Elastic IP) vs what's lost on termination, key pair management 2. S3 operations — bucket creation, object upload/download/list via CLI, the difference between bucket policy and IAM policy for access control, how S3 keys work (no real folders), versioning in one sentence 3. VPC networking — subnets (public vs private, how the route table differs), security groups (stateful, inbound/outbound rules), Internet Gateway, NAT Gateway, when to use each 4. IAM fundamentals — the difference between a user, a role, and a policy; why EC2 instances should use roles not access keys; Instance Profiles; the principle of least privilege as a one-sentence rule 5. AWS CLI pattern — command structure (`aws <service> <action> --<option>`), how to configure a profile, how to use `--query` and `--output` to filter results, the 10 most common commands a DevOps engineer runs daily Write it for my agent to consume. My agent needs to be able to provision infrastructure and debug access issues from this file.
