Bash Scripting Foundation — From Manual Commands to Production-Safe Scripts
How do you turn a sequence of manual commands into a reliable, reusable script that works correctly in all conditions — not just the happy path?
A DevOps engineer who can only run commands interactively is limited to manual operations. Bash is the glue that turns discrete commands into automation: health checks, deployment scripts, backup jobs, log processors. The difference between a bash script that works and one that is production-safe is exactly what this day covers — error handling, input validation, and making the script behave correctly when things go wrong, not just when everything is fine.
The six patterns — what makes a script production-safe
Every production-grade bash script applies all six. A script missing any one of these will eventually fail silently in a way that's hard to diagnose. By EoD, every line of these is in your memory file with concrete syntax.
Pattern 1
Error handling
set -euo pipefail — exit on error, error on undefined vars, fail on pipe errors.
Pattern 2
Input validation
$# for arg count, ${1:?Usage...} for required args, usage function.
Pattern 3
Conditional checks
-f file exists, -d dir exists, -z/-n empty/non-empty. [[ ]] for safety.
Pattern 4
Stdout vs stderr
Errors to >&2, normal output to stdout. 2>/dev/null to silence noise.
Pattern 5
Functions
Definition syntax, return via exit code (not return value), local variables.
Pattern 6
Shebang + PATH hygiene
#!/bin/bash first line, full paths or explicit PATH= for cron/CI safety.
The single most important line
Naive — runs, but silently lies
#!/bin/bash echo "running checks..." bad_command # exits non-zero echo "all good!" # still prints exit 0 # script "succeeds"
Production-safe
#!/bin/bash set -euo pipefail echo "running checks..." bad_command # script exits HERE echo "all good!" # never reached
Discovery chain — 8 questions
1
You write a script that pings an endpoint, checks disk space, then tails an error log. The endpoint is unreachable — the ping hangs 30 seconds and the rest runs anyway. What's wrong, and how do you fix it?
Walk me through error-aware scripting: (1) What is an exit code? Show me how to check the exit code of the previous command (`$?`) and what 0 vs non-zero means. (2) Without `set -e`, what does bash do when a command fails — does the next line still run? Why is that dangerous in a multi-step script? (3) `set -e` — when a command fails, the script exits immediately. Show me the same script with and without `set -e`. (4) Why `set -euo pipefail` is the canonical opening: -e exit on error, -u error on undefined var, -o pipefail makes pipes fail on the first failing command (not just the last). (5) The hang problem: ping waits 30 seconds before timing out. Use `timeout 3 ping -c 1 host` or `ping -W 2 -c 1 host` to bound the wait. Show both. (6) The fixed script — show the diff: shebang, set line, and timeout-wrapped commands.
set -euo pipefail on line 2 of every script. If you only remember one thing from today, this is it.
2
Now set -e kills the script on any failure. But sometimes you want to handle a failure, not exit. How do you distinguish "exit on failure" from "check if it failed and react"?
Teach me the distinction between strict-mode and explicit error handling: (1) Inside an `if` test, `set -e` is suppressed for the test command. So `if ping host; then ...; else ...; fi` works correctly even with -e on. (2) When to use `if cmd; then` vs letting -e exit: - Use if/else when failure is a normal expected branch (file missing, host down → fall back). - Let -e exit when failure means the script can't continue (required tool not installed, required argument missing). (3) Capturing output AND exit code: `if output=$(curl -s --max-time 5 "$URL"); then echo "got: $output"; else echo "fetch failed" >&2; fi`. (4) Force a single command to be allowed to fail: `failing_command || true`. When is this OK and when is it dangerous? (5) Exit code conventions: 0 = success, 1 = generic failure, 2 = misuse (bad args), 64–78 = sysexits.h codes, 126/127 = command not found / not executable.
A common bug: grep pattern file with -e on. If grep finds no match, exit code is 1 — the script dies. Wrap as if grep -q pattern file; then ...; else ...; fi.
3
Your script needs a hostname argument. A teammate runs it with no args — it proceeds against an empty string and produces confusing errors downstream. How do you validate inputs and print a usage message?
Show me the input validation patterns: (1) `$#` is the count of arguments. Show: `if [[ $# -lt 1 ]]; then echo "Usage: $0 " >&2; exit 2; fi`. (2) The `${var:?msg}` pattern: `HOST="${1:?Usage: $0 }"`. If $1 is unset or empty, prints "Usage: ..." to stderr and exits non-zero — replaces the manual check above. (3) The usage function pattern — write a `usage()` function, call it on `--help` or invalid args, exit 2 (misuse). Show me a 5-line usage function. (4) Validating values, not just presence: e.g. require a numeric arg with `[[ $1 =~ ^[0-9]+$ ]] || die "must be numeric"`. Define `die() { echo "$@" >&2; exit 1; }` once. (5) Why exit 2 for bad args (not exit 1)? Convention: 2 = caller misused the script. Lets calling scripts distinguish "tool failed" from "I called it wrong".
Always send error messages to stderr (>&2). When the script's stdout is redirected to a log or piped to another command, error messages still surface to the human.
4
You redirect script output to a log file. curl's progress bar appears mixed in with your real output. What's the difference between stdout and stderr, and how do you separate them?
Stream redirection deep dive: (1) File descriptors: 0 = stdin, 1 = stdout, 2 = stderr. Every process has these three. `>` is shorthand for `1>`, so `cmd > file` redirects stdout only. (2) Common patterns: `cmd > out.log` stdout to file, stderr to terminal `cmd 2> err.log` stderr to file, stdout to terminal `cmd > out.log 2>&1` merge both streams to one file (ORDER MATTERS — see #4) `cmd > out.log 2> err.log` separate files `cmd &> all.log` bash shorthand for both to one file `cmd 2>/dev/null` discard noise, keep stdout (3) curl progress: it goes to stderr by default. Use `curl -s` (silent) to suppress, or `curl -sS` (silent but show errors). (4) Why `>file 2>&1` order matters: 2>&1 means "make 2 point to wherever 1 currently points". If you write `2>&1 >file`, 2 still points to terminal because the redirect happened first. (5) A logging function pattern: log() { echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] $*"; } err() { echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] ERROR: $*" >&2; } Use these instead of bare `echo` everywhere. Timestamps + stderr split for free.
When debugging "why didn't I see the error", check whether the calling code redirected stderr. cmd 2>&1 | tee log is the right shape for logs that capture everything.
5
You parse a log at $LOG_FILE. The file may not exist. How do you check before opening it — and what do -f, -d, -e mean?
Conditional checks for files, strings, and numbers: (1) File tests inside `[[ ]]`: -e exists (any type) -f exists AND is a regular file -d exists AND is a directory -r readable, -w writable, -x executable -s exists AND is non-empty Why prefer -f over -e for log files? Because -e returns true for directories too — if /var/log/app.log is a directory by mistake, your read fails confusingly. (2) String tests: -z str true if empty -n str true if non-empty (default — so `[[ $var ]]` works) == / != string equality (use inside [[ ]], NOT inside [ ]) (3) Number tests: -eq -ne -lt -le -gt -ge. NOT < > = inside [[ ]] for numbers — those are string compares. (4) `[[ ]]` vs `[ ]`: always use `[[ ]]` in bash. Safer (no word splitting), supports regex (=~), supports && and ||. `[ ]` is POSIX-portable but you don't need that for #!/bin/bash. (5) The check-then-act idiom: if [[ -f "$LOG_FILE" ]]; then while read -r line; do ...; done < "$LOG_FILE" else err "log file not found: $LOG_FILE" exit 1 fi
Always quote variable expansions: "$var", not $var. Unquoted variables word-split on whitespace and break on filenames with spaces.
6
Your script is now 80 lines. A teammate reading it has no idea what each section does. Show me the structure of a readable, function-organized bash script.
Bash script structure for readability: (1) Function syntax — all three forms work, prefer the first: check_disk() { df -h | awk '$5+0 > 80 {print}'; } function check_disk { ... } # bash-only, avoid for portability (2) Local variables: inside functions, declare with `local var=value`. Otherwise the variable leaks to global scope and your refactor breaks distantly. (3) Functions return via EXIT CODE, not a "return value". `return 0` is success, `return 1` is failure. To capture output, use `result=$(my_func arg)`. (4) Standard structure: 1. Shebang + set -euo pipefail 2. Constants (uppercase by convention): readonly LOG_FILE=/var/log/app.log 3. Helper functions: log(), err(), die(), usage() 4. Domain functions: check_disk(), check_endpoint(), parse_log() 5. main() function — orchestrates everything 6. Last line: main "$@" — pass all script args to main (5) Comments: section headers and WHY, not WHAT. Don't comment `# get disk usage` above an obvious df command. Do comment `# Server is in PRC, must use TZ=Asia/Shanghai for log timestamps`.
Run shellcheck script.sh — a static analyzer that catches 90% of bash bugs before runtime. Install via brew install shellcheck. Make it a habit before every commit.
Use readonly for values that should never change: readonly DEPLOY_USER=deploy. Catches accidental reassignment.
7
Script works locally but fails in CI with jq: command not found. What's PATH, why is it different in CI vs your shell — and how do you make scripts work regardless?
PATH and environment hygiene for cron/CI: (1) `echo $PATH` in interactive shell: long, includes /opt/homebrew/bin, ~/bin, asdf shims, etc. — populated by your shell rc files (~/.bashrc, ~/.zshrc). (2) `cron` and CI runners use a minimal PATH (usually `/usr/bin:/bin`). Your shell rc files DON'T run. So `jq` installed in /opt/homebrew/bin is invisible. (3) Three fixes ranked from worst to best: - Worst: hardcode full path: `/opt/homebrew/bin/jq` — breaks when teammate has Linux paths. - Middle: set PATH at top of script: `export PATH="/usr/local/bin:/usr/bin:/bin:$PATH"`. - Best: explicitly check at script start that required tools exist: for cmd in jq curl aws; do command -v "$cmd" >/dev/null || die "missing required tool: $cmd"; done (4) `command -v` is the modern replacement for `which` — POSIX-portable, exit code is reliable. (5) Other env quirks in cron: no TERM, no HOME shortcut, working directory is $HOME. If your script does `cd somewhere` first, it works the same in cron and shell. (6) For CI, also ensure: container image has the required tools installed (apt-get install jq in Dockerfile or setup step), or use an image that bundles them.
Add a "preflight" function at the top of every nontrivial script that checks every required command is on PATH. Five seconds of boilerplate saves an hour of "why is it failing only in CI".
