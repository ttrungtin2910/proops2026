Docker Core: Dockerfile, Image, Build, Run, Exec
How do you take an application that runs on one specific machine and make it run identically everywhere — on a laptop, a CI server, and a production cluster?
"It works on my machine" is the defining problem Docker solves. Every modern deployment pipeline produces a Docker image. Every Kubernetes workload runs a Docker container. A DevOps engineer who cannot read, write, build, and debug a Dockerfile is missing the foundational skill for container-based operations — which is now the default for almost every production system.

Discovery Chain — 8 questions, one containerization scenario
Today's 8 questions follow a single scenario — you receive a Node.js application and must containerize it, debug it, optimize it, and configure it correctly. Copy each purple prompt, paste into Claude, run the commands, observe the output, then move on.
1
A developer hands you a Node.js app that runs locally. Without Docker, what five things must be identically configured on every machine — and how does this scale as the team grows?
I'm a DevOps engineer. A developer hands me a Node.js application that "works on their machine." Walk me through: (1) the five things that must be identically configured on every machine for this app to run — Node.js version, package dependencies (node_modules), environment variables, config file paths, and OS-level libraries — and why each one is a source of environment drift, (2) what happens to this list as the team grows from 5 to 50 engineers — how many combinations of OS + Node version + package manager you're now managing, (3) what "environment parity" means and why it is the core DevOps problem Docker solves. Concrete examples, not theory.
Environment drift is the root cause. Docker solves it by making the environment part of the artifact.
2
You write your first Dockerfile. The COPY command copies node_modules (400MB) every build. What is .dockerignore — and what is the build context?
I'm writing my first Dockerfile for a Node.js app and every build sends 400MB of node_modules to the Docker daemon. Explain: (1) what the "build context" is — exactly what gets sent to the Docker daemon when you run `docker build`, why it matters, and how to check its size, (2) what a .dockerignore file is, how it works (same syntax as .gitignore), and what I must put in it for a Node.js project, (3) why copying a 400MB node_modules directory is wrong even beyond the size — why I should never copy node_modules from the host into the image, (4) show me the correct Dockerfile for a Node.js app: the right base image, the right order of COPY and RUN npm install, and the correct .dockerignore. Give me the actual files, not pseudocode.
Build context = everything in the directory. .dockerignore = what to exclude. node_modules belongs inside the image, not copied from host.
3
You change one line in app.js and rebuild — it takes 3 minutes again. What is the Docker layer cache, why is it not working, and how do you fix the Dockerfile?
I built my Docker image in 3 minutes. I changed one line in app.js and rebuilt — it took 3 minutes again. Explain: (1) what the Docker layer cache is — how each Dockerfile instruction creates a layer, how layers are cached by comparing the instruction + context, and what "cache invalidation" means in this context, (2) why changing app.js breaks the cache for ALL subsequent layers including npm install — trace exactly what happens in a broken Dockerfile order, (3) the "dependencies before code" rule — show me the two-step COPY pattern that makes npm install cache-stable: `COPY package*.json ./` then `RUN npm install` then `COPY . .`, (4) show me the before and after Dockerfile side by side with a comment explaining why the order matters. I want to see the exact instruction sequence that makes a one-line app.js change rebuild in under 5 seconds instead of 3 minutes.
Cache breaks at the first changed layer and invalidates everything below. Copy package.json first, code second.
4
You run docker run myapp and the container exits immediately with no error message. What three commands tell you what happened?
I ran `docker run myapp` and the container exits immediately. There is no error message in the terminal. Walk me through: (1) `docker ps -a` — what this shows (all containers including stopped ones), how to read the STATUS and EXITED columns, and what exit code 1 vs exit code 0 means, (2) `docker logs [container-id]` — how to get the container's stdout/stderr output after it exits, what to look for, and why the container's output goes to logs not the terminal when running in the background, (3) `docker inspect [container-id]` — how to read the State section to find the exit code and the error, (4) the three most common reasons a container exits immediately (wrong CMD, missing environment variable, missing file) and how to diagnose each one. Give me the exact commands and what to look for in the output.
docker ps -a → docker logs → docker inspect. Always check logs before guessing.
5
The app is running but you can't reach it from your browser. It listens on port 3000 inside the container. What does -p 3000:3000 do — and why is the container's port not accessible without it?
My Node.js app is running in a container (`docker ps` shows it running) but I can't reach http://localhost:3000 from my browser. The app listens on port 3000 inside the container. Explain: (1) what Docker's network isolation model means — why each container has its own network namespace, its own localhost, and its own port space, so port 3000 inside the container is NOT the same as port 3000 on my host, (2) what `-p 3000:3000` does — the HOST_PORT:CONTAINER_PORT format, how Docker sets up NAT rules to forward traffic, and what happens if I use `-p 8080:3000`, (3) the difference between `EXPOSE` in the Dockerfile and `-p` in `docker run` — what EXPOSE actually does (documentation only, does NOT publish the port), (4) what `docker port [container-id]` shows and how to verify the port mapping is active. Show me the exact commands.
Container network = isolated namespace. -p = NAT rule from host port to container port. EXPOSE is documentation, not a firewall rule.
6
The database URL is hardcoded in the Dockerfile. What is the right way to pass runtime config to a container — and why is a hardcoded secret a permanent problem even after deletion?
I notice the database URL is hardcoded as `ENV DB_URL=postgres://user:password@prod-db:5432/app` in my Dockerfile. Explain: (1) exactly why this is a permanent security problem — how Docker image layers work, why removing the ENV instruction in a new layer does NOT remove it from the image history, and what `docker history --no-trunc` reveals, (2) the correct pattern for runtime configuration: passing environment variables via `docker run -e DB_URL=...` or `--env-file .env`, (3) what an env file looks like, how to use it with docker run, and why the .env file must be in .gitignore and .dockerignore, (4) the difference between build-time ARG and runtime ENV — when to use each, (5) for secrets specifically (passwords, API keys): why env vars are not ideal either, and what Docker secrets or mounted secret files solve. Show me the Dockerfile, the docker run command, and the env file format.
Image layers are permanent. A secret baked in is a secret exposed — forever in the image history. Use -e or --env-file at runtime.
7
A bug is occurring inside the running container. You need to examine the filesystem and processes. What is docker exec — and how is it different from SSH?
A bug is occurring inside a running container and I need to debug it. I tried SSHing in but there is no SSH daemon running. Explain: (1) what `docker exec -it [container-id] /bin/sh` does — how exec attaches a new process to an already-running container's namespaces (PID, network, filesystem), why it doesn't need SSH, (2) what I can do once I'm inside: check the filesystem (`ls -la /app`), view running processes (`ps aux`), check environment variables (`env`), test network connectivity (`curl`, `wget`), inspect logs (`cat /var/log/app.log`), (3) the difference between `docker exec` and `docker run` — exec attaches to a running container, run starts a new one, (4) why production containers often don't have bash installed (Alpine base images use `/bin/sh`, not `/bin/bash`) and what to do when even sh is missing, (5) what `docker cp` does for extracting files from a container without exec. Give me the exact commands for a typical debug session.
exec = new process inside existing container namespaces. No SSH needed. Alpine uses /bin/sh not /bin/bash.
8
Write memory/docker.md — the five Docker concepts every DevOps engineer must explain and use without thinking.
I'm writing memory/docker.md for my DevOps agent. Generate the full file with these 5 sections: 1. Dockerfile anatomy — FROM, RUN, COPY, WORKDIR, ENV, EXPOSE, CMD, ENTRYPOINT: what each instruction does, a one-line rule for when to use it, and a complete working Dockerfile example for a Node.js app that demonstrates the "dependencies before code" layer cache pattern 2. Layer cache — how Docker caches each instruction as a layer, what triggers cache invalidation, the correct instruction order for stable cache, and how to verify cache hits during a build (look for "CACHED" in output) 3. Core commands — build, run, exec, logs, ps, stop, rm, rmi, pull, tag: the key flags for each (-d, -p, -e, --name, -it, --rm, -f, --no-cache), with a one-line example for each command showing the most common real-world usage 4. Port mapping and network isolation — the HOST:CONTAINER format, what "container has its own network namespace" means in one sentence, the difference between EXPOSE and -p, how to check active port mappings 5. Image vs container distinction — an image is a blueprint (class / ISO file), a container is a running instance (object / running VM): the analogy, what `docker images` vs `docker ps` shows, and the relationship between image layers and container filesystem (copy-on-write layer) Write it for my agent to act on. My agent needs to write Dockerfiles, debug containers, and explain Docker concepts to developers.
